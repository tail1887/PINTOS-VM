# Pintos 1주차 시작 전 코치 강의 정리

코치 강의에서 들은 내용을 바탕으로, 구현 시작 전에 팀이 공통으로 맞춰야 할 핵심 포인트를 정리한 문서입니다.

## 1) 큰 흐름: 전원 ON부터 커널 코드 실행까지

- 컴퓨터 부팅 흐름은 보통 `Firmware(BIOS/UEFI) -> Bootloader -> OS Kernel` 순서다.
- Pintos에서는 커널 초기화가 진행되며, 실질적인 시작 지점은 `threads/init.c`의 `main()`이다.
- 즉, "내가 작성한 코드가 언제부터 실행되는가?"를 이해하려면 `init.c`부터 보는 것이 가장 안전하다.

## 2) CPU와 스레드, 그리고 스케줄링

- CPU는 한 번에 하나의 실행 흐름만 직접 수행한다.
- 여러 작업이 동시에 되는 것처럼 보이는 이유는 OS가 짧은 tick 단위로 실행 대상을 바꾸기 때문이다.
- 이때 문맥 교환(Context Switching) 비용이 커지면 전체 성능이 떨어진다.
- 특히 인터럽트 핸들러는 짧고 빠르게 끝나야 한다.

## 3) 인터럽트 처리에서 주의할 점

- 인터럽트 처리 중에는 임계 구역 보호가 필요할 수 있다.
- 공유 자료구조를 건드릴 때는 `intr_disable()` 같은 방식으로 원자성을 보장해야 한다.
- 단, 인터럽트를 오래 막으면 지연이 커지므로 "필요한 최소 구간"만 보호하는 습관이 중요하다.

## 4) Mission 1: Alarm Clock 핵심

### 문제 의식

- 기본 `timer_sleep()`은 busy waiting(반복 확인)으로 CPU를 낭비한다.
- "while로 계속 시간 확인"하는 구조를 제거하는 것이 목표다.

### 구현 방향

- 잠들 스레드를 `ready_list`에서 빼서 별도 `sleep_list`로 이동시킨다.
- 스레드 상태를 block 상태로 전환해 스케줄링 대상에서 제외한다.
- 타이머 인터럽트가 발생할 때마다 깨울 시간이 된 스레드만 다시 ready 상태로 복귀시킨다.

### 확인할 파일

- 타이머 인터럽트 핸들러: `devices/timer.c`
- 커널 시작점: `threads/init.c`
- 스레드 관련 구현: `threads/` 하위 코드

## 5) Mission 2: Priority Scheduling

- 우선순위 기반 스케줄링은 ready 큐 관리 방식이 핵심이다.
- 삽입 시 정렬을 유지하면 선택 비용이 줄어든다.
- 꼭 기억할 문제는 Priority Inversion(우선순위 역전)이다.
- 높은 우선순위 스레드가 낮은 우선순위 스레드의 lock을 기다리며 막히는 현상을 해결하려면 Priority Donation이 필요하다.

## 6) Mission 3: MLFQS 개념

- MLFQS(Multi-Level Feedback Queue Scheduler)는 스레드의 CPU 사용 패턴을 반영해 동적으로 우선순위를 조정한다.
- CPU를 많이 점유한 스레드는 우선순위를 낮추고, I/O 친화적인 스레드는 상대적으로 유리해질 수 있다.
- Pintos에서는 보통 고정소수점 기반 계산으로 스케줄링 값을 갱신한다.

## 7) 테스트/빌드 운영 방법

- 구현 후 `threads/`에서 빌드하고 테스트를 수행한다.
- 전체 테스트 전에 단일 테스트 케이스부터 돌리며 검증하는 것이 효율적이다.
- 예: alarm 계열부터 시작해 점진적으로 범위를 넓힌다.
- 테스트는 실행 결과를 로그/기대 결과와 비교해 PASS/FAIL을 판단한다.
- 따라서 "코드 작성"만큼 "테스트가 무엇을 검증하는지 이해"가 중요하다.

## 8) 협업 전략 (팀 프로젝트 관점)

- 팀 브랜치 + 개인 브랜치 전략으로 작업 분리 후 PR 기반으로 합친다.
- 테스트 케이스 단위로 역할 분담하면 병렬 진행이 쉽다.
- 머지 후 통합 테스트를 반드시 다시 돌려야 한다.
- 코드리뷰 시간에 구현 의도와 근거를 설명할 수 있어야 팀 속도가 올라간다.

## 9) 실전 팁

- Pintos의 기본 라이브러리(`lib/kernel/list.c` 등)를 적극 활용한다.
- 자료구조를 새로 만들기보다 기존 유틸을 정확히 사용하는 것이 안전하다.
- 디버깅은 GDB가 가장 좋지만, 필요하면 `printf`로 실행 흐름을 좁혀도 된다.
- "모든 테스트를 억지로 다 통과"보다 "통과한 이유를 설명 가능"한 구현이 더 중요하다.

## 10) 1주차 학습 목표 정리

- Alarm Clock의 busy waiting 제거를 직접 구현할 수 있다.
- 우선순위 스케줄링/기부의 필요성을 설명할 수 있다.
- 타이머 인터럽트와 스레드 상태 전이를 코드 레벨로 추적할 수 있다.
- 테스트 케이스를 읽고, 어떤 동작을 요구하는지 해석할 수 있다.
