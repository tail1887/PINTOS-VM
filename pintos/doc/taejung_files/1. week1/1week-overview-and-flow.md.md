# Pintos 실행 환경 기초 흐름 정리

이 문서는 "컴퓨터가 켜진 뒤 내가 작성한 Pintos 코드가 실행되기까지"를 단계별로 연결해 설명합니다.  
핵심은 각 단계의 역할과, 다음 단계로 왜 넘어가야 하는지를 이해하는 것입니다.

## 1) 컴퓨터가 실행될 때 소프트웨어 순서

기본 순서:

`펌웨어(BIOS/UEFI) -> 부트로더(Bootloader) -> OS 커널(Pintos)`

### 왜 이 순서가 필요한가?

- 전원 직후 CPU는 "무엇을 실행할지" 모른다.
- 그래서 가장 먼저 펌웨어가 하드웨어를 점검하고, 다음 실행 대상(부트로더)을 찾는다.
- 부트로더는 커널 이미지를 메모리에 올리고 실행 가능한 상태를 만든다.
- 마지막으로 커널이 자원 관리(CPU/메모리/디스크) 주도권을 가져간다.

### 지난 단계와의 연결성

- 펌웨어가 없으면 부트로더를 찾을 수 없다.
- 부트로더가 없으면 커널을 메모리에 정확히 적재할 수 없다.
- 커널이 올라와야 사용자 레벨 프로그램 실행 기반이 생긴다.

## 2) Pintos는 무엇인가?

- Pintos는 운영체제 핵심 원리를 학습하기 위한 교육용 커널 프로젝트다.
- 실무 OS(리눅스/윈도우)보다 작고 구조가 명확해 스케줄링, 동기화, 가상메모리, 파일시스템을 단계적으로 구현할 수 있다.

### 왜 알아야 하는가?

- "OS를 사용"하는 게 아니라 "OS를 직접 구현"하는 학습이기 때문이다.
- 따라서 단순 API 사용이 아니라 커널 내부 자료구조와 제어 흐름을 이해해야 한다.

### 연결성

- 위의 부팅 흐름에서 마지막 단계(OS 커널)에 해당하는 실습 대상이 바로 Pintos다.
- 1주차에서는 특히 `threads` 계층을 다루며, "CPU 시간을 어떤 규칙으로 나눌지"를 코드로 구현하게 된다.

## 3) QEMU란 무엇인가?

- QEMU는 하드웨어를 소프트웨어로 모사하는 에뮬레이터(가상 머신 실행기)다.
- Pintos 커널 이미지를 실제 장비 대신 가상 컴퓨터에서 안전하게 실행/디버깅할 수 있게 해준다.

### 왜 필요한가?

- 커널 코드는 시스템 전체를 제어하므로 오류가 나면 실제 장비가 불안정해질 수 있다.
- QEMU 환경에서는 실패를 격리하고 반복 실험하기 쉽다.

### 연결성

- GCC로 빌드한 커널 바이너리를 "실행해볼 대상"이 필요하고, 그 역할을 QEMU가 담당한다.

## 4) OS가 C 프로그램이라면 시작점은 무엇인가?

- C 관점의 논리 시작점은 `main()`이다.
- Pintos에서는 `threads/init.c`의 `main()`이 핵심 진입점이다.
- 다만 그 전에 아주 초반 부트스트랩 어셈블리 코드가 선행되어 CPU/스택/초기 환경을 세팅한다.

### 왜 이 구분이 중요한가?

- "전원 ON 직후 바로 C가 실행된다"가 아니라,
- "초기화(어셈블리) -> C 초기화(main)" 순서로 올라온다는 점을 알아야 디버깅 시점이 명확해진다.

### 연결성

- 부트로더가 커널 엔트리로 점프하면,
- 초기 부트스트랩을 거쳐 최종적으로 `main()`으로 흐름이 이어진다.

## 4-1) `main()` 이후 1주차 코드와 직접 연결되는 흐름

- `main()`은 커널 전역 초기화를 수행한 뒤, 스레드 시스템을 준비한다.
- 이 시점부터 "어떤 스레드를 언제 실행시킬지"가 운영체제의 핵심 제어 로직이 된다.
- 1주차 과제는 이 스레드 스케줄링 경로에 직접 개입한다.

### 왜 중요할까?

- Alarm Clock은 "실행 중 스레드를 잠재웠다가 다시 ready 상태로 복귀"시키는 문제다.
- Priority Scheduling은 "ready 상태 스레드 중 누가 먼저 CPU를 받을지"를 정의한다.
- 즉, 부팅 지식은 단순 배경이 아니라 과제 코드의 실행 맥락 자체다.

## 5) QEMU는 어떤 시뮬레이터로, 어떤 컴파일러와 연결되는가?

- QEMU는 x86-64 같은 아키텍처를 포함해 시스템 전체(CPU, 타이머, 인터럽트, 메모리 장치 등)를 에뮬레이션할 수 있다.
- Pintos 과제에서는 보통 x86-64 타깃으로 실행한다.
- 컴파일러는 일반적으로 GCC를 사용한다.

### 왜 GCC를 쓰는가?

- C 코드를 타깃 아키텍처(64비트 x86)에 맞는 기계어로 변환해야 하기 때문이다.
- 커널 빌드에 필요한 옵션/도구 체인과 궁합이 좋고, 과제 환경과 문서가 GCC 기준으로 맞춰져 있다.

### 연결성

- 소스 코드(C) -> GCC 컴파일/링크 -> 커널 이미지 생성 -> QEMU에서 부팅/실행
- 즉, GCC는 "만드는 도구", QEMU는 "실행/검증하는 장비" 역할이다.

## 5-1) 빌드 결과물과 실행의 연결

- `threads`에서 빌드하면 커널 실행 이미지가 만들어진다.
- QEMU는 이 이미지를 로드해 가상 하드웨어에서 부팅 과정을 재현한다.
- 테스트는 이 실행 결과를 기준 출력과 비교해 PASS/FAIL을 판정한다.

### 왜 알아야 하나?

- "코드를 수정 -> 빌드 -> QEMU 실행 -> 테스트 비교"가 1주차의 기본 개발 루프이기 때문이다.
- 문제 발생 시, 수정 위치(코드)와 관찰 지점(테스트 로그)을 빠르게 연결할 수 있다.

## 6) 이번 환경은 64비트인가?

- 네, 이번 과정은 64비트(x86-64) 기준으로 이해하면 된다.

### 왜 중요할까?

- 레지스터 폭, 호출 규약, 주소 공간, 자료형 크기/정렬 등에서 32비트와 차이가 난다.
- 스레드 컨텍스트 저장/복원, 인터럽트 처리, 포인터 연산 이해에 직접 영향이 있다.

### 연결성

- GCC가 64비트 타깃 바이너리를 만들고,
- QEMU가 해당 아키텍처 환경을 에뮬레이션하며,
- Pintos 커널 코드는 그 위에서 동작한다.

## 6-1) 1주차 범위 경계: `threads`와 `userprog`

- 1주차의 주된 실행/검증 대상은 `threads`다.
- `userprog`는 다음 주차 주제(사용자 프로그램 실행/시스템콜)와 연결된다.

### 왜 경계를 알아야 하나?

- 현재 단계에서 `userprog`까지 동시에 파고들면 학습 초점이 흐려진다.
- 먼저 스레드/스케줄링/인터럽트 흐름을 확실히 잡아야 다음 주차의 사용자-커널 경계를 이해하기 쉽다.

## 7) 한 번에 연결해서 보기

1. 전원 ON
2. 펌웨어가 초기 점검 + 부트로더 로드
3. 부트로더가 Pintos 커널 이미지를 메모리에 적재
4. 부트스트랩 코드가 초기 환경 세팅
5. `threads/init.c`의 `main()` 진입
6. 커널 초기화 후 스레드/인터럽트 기반 스케줄링 시작
7. `timer_sleep()`/ready 큐/우선순위 로직 수정
8. 빌드 + QEMU 실행 + 테스트 비교로 검증
9. 이후 범위를 넓혀 `userprog` 단계로 확장

## 짧은 결론

Pintos 학습의 출발점은 "코드를 어디서 고칠까?"보다  
"내 코드가 어떤 부팅/실행 체인 위에서 돌아가는가?"를 아는 것입니다.  
이 흐름이 잡히면 각 미션(알람, 우선순위, MLFQS)의 목적과 위치가 훨씬 선명해집니다.
