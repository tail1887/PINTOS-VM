이 섹션에서는 memory-mapped page를 구현합니다. 익명 페이지와 달리 memory-mapped page는 file-backed mapping입니다. 페이지 안의 내용은 기존 파일의 데이터를 반영합니다. 페이지 폴트가 발생하면 즉시 물리 프레임을 할당하고 파일에서 내용을 메모리로 복사합니다. memory-mapped page가 unmap되거나 swap out될 때는 내용의 변경 사항이 파일에 반영됩니다.

#### `mmap` 및 `munmap` 시스템 콜

**memory mapped file을 위한 두 시스템 콜인 `mmap`과 `munmap`을 구현하세요.** VM 시스템은 mmap 영역의 페이지를 lazy loading으로 적재해야 하며, mmap된 파일 자체를 해당 mapping의 backing store로 사용해야 합니다. 이 두 시스템 콜을 구현하기 위해 `vm/file.c`에 정의된 `do_mmap`과 `do_munmap`을 구현하고 사용해야 합니다.

---

```
void *mmap (void *addr, size_t length, int writable, int fd, off_t offset);
```

> `fd`로 열린 파일의 `offset` 바이트부터 시작하여 `length` 바이트를 프로세스의 가상 주소 공간 `addr`에 매핑합니다. 전체 파일은 `addr`에서 시작하는 연속된 가상 페이지들에 매핑됩니다. 파일 길이가 PGSIZE의 배수가 아니라면 마지막 매핑 페이지의 일부 바이트는 파일 끝을 넘어 "삐져나옵니다". 페이지가 fault in될 때 이 바이트들을 0으로 설정하고, 디스크에 다시 쓸 때는 버립니다. 성공하면 이 함수는 파일이 매핑된 가상 주소를 반환합니다. 실패하면 파일을 매핑할 수 있는 유효한 주소가 아닌 NULL을 반환해야 합니다.

`fd`로 열린 파일의 길이가 0바이트라면 `mmap` 호출은 실패할 수 있습니다. `addr`이 page-aligned가 아니거나, 매핑되는 페이지 범위가 스택 또는 실행 파일 로드 시 매핑된 페이지를 포함한 기존 매핑 페이지 집합과 겹치면 반드시 실패해야 합니다. Linux에서는 `addr`이 NULL이면 커널이 mapping을 만들 적절한 주소를 찾습니다. 단순화를 위해, 여러분은 주어진 `addr`에 mmap을 시도하면 됩니다. 따라서 `addr`이 0이면 실패해야 합니다. 일부 Pintos 코드는 가상 페이지 0이 매핑되지 않았다고 가정하기 때문입니다. 또한 `length`가 0일 때도 mmap은 실패해야 합니다. 마지막으로 콘솔 입력과 출력을 나타내는 파일 디스크립터는 매핑할 수 없습니다.

Memory-mapped page도 익명 페이지처럼 lazy 방식으로 할당해야 합니다. 페이지 객체를 만들기 위해 `vm_alloc_page_with_initializer` 또는 `vm_alloc_page`를 사용할 수 있습니다.

---

```
void munmap (void *addr);
```

> 지정된 주소 범위 `addr`에 대한 mapping을 해제합니다. `addr`은 같은 프로세스가 이전에 mmap 호출로 반환받았고 아직 unmap하지 않은 가상 주소여야 합니다.

모든 mapping은 프로세스가 `exit`을 통해 종료되든 다른 방식으로 종료되든 프로세스 종료 시 암묵적으로 unmap됩니다. mapping이 명시적으로든 암묵적으로든 unmap될 때, 프로세스가 쓴 모든 페이지는 파일에 다시 기록되어야 하며, 쓰지 않은 페이지는 기록되어서는 안 됩니다. 그다음 페이지들은 프로세스의 가상 페이지 목록에서 제거됩니다.

파일을 닫거나 삭제해도 해당 파일의 mapping은 unmap되지 않습니다. Unix 관례에 따라 한 번 생성된 mapping은 `munmap`이 호출되거나 프로세스가 종료될 때까지 유효합니다. 자세한 내용은 [Removing an Open File](https://casys-kaist.github.io/pintos-kaist/project2/FAQ.html)을 참고하세요. 각 mapping마다 파일에 대한 별도의 독립적인 참조를 얻기 위해 `file_reopen` 함수를 사용해야 합니다.

두 개 이상의 프로세스가 같은 파일을 매핑하더라도, 서로 일관된 데이터를 보아야 한다는 요구사항은 없습니다. Unix는 두 mapping이 같은 물리 페이지를 공유하도록 하여 이를 처리하며, mmap 시스템 콜에는 클라이언트가 페이지를 shared로 할지 private, 즉 copy-on-write로 할지 지정할 수 있는 인자도 있습니다.

필요에 따라 `vm/vm.c`의 `vm_file_init`과 `vm_file_initializer`를 수정할 수 있습니다.

---

```
void vm_file_init (void);
```

> file-backed page 서브시스템을 초기화합니다. 이 함수 안에서 file-backed page와 관련된 모든 것을 설정할 수 있습니다.

---

```
bool file_backed_initializer (struct page *page, enum vm_type type, void *kva);
```

> file-backed page를 초기화합니다. 이 함수는 먼저 `page->operations`에 file-backed page용 handler를 설정합니다. 메모리를 backing하는 파일 등 page struct의 정보를 갱신해야 할 수 있습니다.

---

```
static void file_backed_destroy (struct page *page);
```

> 연결된 파일을 닫아 file-backed page를 파괴합니다. 내용이 dirty라면 변경 사항을 파일에 다시 기록해야 합니다. 이 함수 안에서 page struct를 해제할 필요는 없습니다. `file_backed_destroy`의 호출자가 이를 처리해야 합니다.
