가상 메모리 시스템을 지원하려면 가상 페이지와 물리 프레임을 효과적으로 관리해야 합니다. 즉 어떤 가상 또는 물리 메모리 영역이 사용 중인지, 어떤 목적으로 쓰이는지, 누가 사용하는지 등을 추적해야 합니다. 먼저 supplemental page table을 다루고, 그다음 물리 프레임을 다룹니다. 이해를 돕기 위해 이 문서에서는 가상 페이지를 "page"라고 부르고, 물리 페이지를 "frame"이라고 부릅니다.

### Page Structure와 Operations

#### struct page

`include/vm/vm.h`에 정의된 `page`는 가상 메모리 안의 페이지를 나타내는 구조체입니다. 페이지에 대해 알아야 하는 필요한 모든 데이터를 저장합니다. 현재 템플릿에서 이 구조체는 다음과 같습니다.

```
struct page {
  const struct page_operations *operations;
  void *va;              /* 사용자 공간 기준 주소 */
  struct frame *frame;   /* frame에 대한 역참조 */

  union {
    struct uninit_page uninit;
    struct anon_page anon;
    struct file_page file;
#ifdef EFILESYS
    struct page_cache page_cache;
#endif
  };
};
```

이 구조체는 page operation(아래 설명), 가상 주소, 물리 프레임을 가집니다. 추가로 union 필드도 있습니다. union은 하나의 메모리 영역에 서로 다른 타입의 데이터를 저장할 수 있게 하는 특수한 자료형입니다. union 안에는 여러 멤버가 있지만, 한 번에 하나의 멤버만 값을 가질 수 있습니다. 이는 우리 시스템의 페이지가 uninit_page, anon_page, file_page, 또는 page_cache 중 하나일 수 있음을 뜻합니다. 예를 들어 어떤 페이지가 익명 페이지라면 [Anonymous Page](https://casys-kaist.github.io/pintos-kaist/project3/anon.html)를 참고하세요. page struct는 멤버 중 하나로 `struct anon_page anon` 필드를 가지게 됩니다. `anon_page`는 익명 페이지에 대해 보관해야 하는 모든 필요한 정보를 담습니다.

#### Page Operations

위에서 설명했고 `include/vm/vm.h`에 정의되어 있듯이, 페이지는 `VM_UNINIT`, `VM_ANON`, 또는 `VM_FILE`일 수 있습니다. 페이지에 대해서는 swap in, swap out, destroy 같은 여러 동작이 수행됩니다. 각 페이지 타입마다 이러한 동작에 필요한 단계와 작업이 다릅니다. 다시 말해 `VM_ANON` 페이지와 `VM_FILE` 페이지에는 서로 다른 `destroy` 함수가 호출되어야 합니다. 한 가지 방법은 각 함수에서 switch-case 문법으로 각 경우를 처리하는 것입니다. 여기서는 이를 다루기 위해 객체 지향 프로그래밍의 "class inheritance" 개념을 도입합니다. C 언어에는 실제로 "class"도 "inheritance"도 없지만, Linux 같은 실제 운영체제 코드와 비슷한 방식으로 [function pointer](https://www.geeksforgeeks.org/function-pointer-in-c/)를 사용해 이 개념을 구현합니다.

함수 포인터는 여러분이 지금까지 배운 다른 포인터처럼 포인터이지만, 함수 또는 메모리 안의 실행 가능한 코드를 가리킵니다. 함수 포인터는 런타임 값에 따라 특정 함수를 검사 없이 호출할 수 있는 간단한 방법을 제공하므로 유용합니다. 우리의 경우 코드 수준에서는 단순히 `destroy (page)`를 호출하면 충분하고, 컴파일러는 올바른 함수 포인터를 호출해 페이지 타입에 맞는 `destroy` 루틴을 선택합니다.

page operation 구조체인 `struct page_operations`는 `include/vm/vm.h`에 정의되어 있습니다. 이 구조체를 3개의 함수 포인터를 담은 함수 테이블로 생각하면 됩니다.

```
struct page_operations {
  bool (*swap_in) (struct page *, void *);
  bool (*swap_out) (struct page *);
  void (*destroy) (struct page *);
  enum vm_type type;
};
```

이제 page_operation 구조체를 어디서 찾을 수 있는지 보겠습니다. `include/vm/vm.h`의 page 구조체 `struct page`를 보면 `operations`라는 필드가 있습니다. 이제 `vm/file.c`로 가면 함수 프로토타입 앞에 선언된 page_operations 구조체 `file_ops`를 볼 수 있습니다. 이것은 file-backed page를 위한 함수 포인터 테이블입니다. `.destroy` 필드는 `file_backed_destroy` 값을 가지며, 이는 페이지를 파괴하는 함수로 같은 파일에 정의되어 있습니다.

함수 포인터 인터페이스로 `file_backed_destroy`가 어떻게 호출되는지 이해해봅시다. `vm/vm.c`의 `vm_dealloc_page (page)`가 호출되었고, 이 페이지가 file-backed page(`VM_FILE`)라고 가정합니다. 함수 내부에서는 `destroy (page)`를 호출합니다. `destroy (page)`는 `include/vm/vm.h`에서 다음과 같은 매크로로 정의되어 있습니다.

```
#define destroy(page) if ((page)->operations->destroy) (page)->operations->destroy (page)
```

이는 `destroy` 함수를 호출하면 실제로는 page 구조체에서 가져온 `destroy` 함수인 `(page)->operations->destroy (page)`가 호출된다는 것을 알려줍니다. 페이지가 `VM_FILE` 페이지이므로, 그 `.destroy` 필드는 `file_backed_destory`를 가리킵니다. 결과적으로 file-backed page에 대한 destroy routine이 수행됩니다.

### Supplemental Page Table 구현

현재 Pintos에는 메모리의 가상-물리 mapping을 관리하는 페이지 테이블(`pml4`)이 있습니다. 하지만 이것만으로는 충분하지 않습니다. 이전 섹션에서 논의했듯이, page fault와 자원 관리를 처리하기 위해 각 페이지에 대한 추가 정보를 담는 supplementary page table도 필요합니다. 따라서 프로젝트 3의 첫 번째 과제로 supplemental page table의 기본 기능 구현을 제안합니다.

**`vm/vm.c`에 supplemental page table 관리 함수를 구현하세요.**

먼저 Pintos에서 supplemental page table을 어떻게 설계할지 결정해야 합니다. 자신의 supplemental page table을 설계한 뒤, 그 설계에 맞춰 아래 세 함수를 구현하세요.

---

```
void supplemental_page_table_init (struct supplemental_page_table *spt);
```

> supplemental page table을 초기화합니다. supplemental page table에 사용할 자료구조는 자유롭게 선택할 수 있습니다. 이 함수는 새 프로세스가 시작될 때(`userprog/process.c`의 `initd`)와 프로세스가 fork될 때(`userprog/process.c`의 `__do_fork`) 호출됩니다.

---

```
struct page *spt_find_page (struct supplemental_page_table *spt, void *va);
```

> 주어진 supplemental page table에서 va에 대응하는 `struct page`를 찾습니다. 실패하면 NULL을 반환합니다.

---

```
bool spt_insert_page (struct supplemental_page_table *spt, struct page *page);
```

> 주어진 supplemental page table에 `struct page`를 삽입합니다. 이 함수는 해당 가상 주소가 주어진 supplemental page table에 이미 존재하지 않는지 검사해야 합니다.

### Frame Management

이제부터 모든 페이지는 생성 시점의 메모리 metadata만 들고 있는 것이 아닙니다. 따라서 물리 메모리를 관리하기 위한 다른 체계가 필요합니다. `include/vm/vm.h`에는 물리 메모리를 나타내는 `struct frame`이 있습니다. 현재 구조체는 다음과 같습니다.

```
/* "frame"의 표현 */
struct frame {
  void *kva;
  struct page *page;
};
```

이 구조체에는 두 필드만 있습니다. `kva`는 커널 가상 주소이고, `page`는 page 구조체입니다. frame management interface를 구현하면서 멤버를 더 추가해도 됩니다.

**`vm/vm.c`에 `vm_get_frame`, `vm_claim_page`, `vm_do_claim_page`를 구현하세요.**

---

```
static struct frame *vm_get_frame (void);
```

> `palloc_get_page`를 호출해 user pool에서 새 물리 페이지를 얻습니다. user pool에서 페이지를 성공적으로 얻으면 frame도 할당하고, 그 멤버를 초기화한 뒤 반환합니다. `vm_get_frame`을 구현한 뒤에는 모든 user space page(PALLOC_USER)를 이 함수를 통해 할당해야 합니다. 지금은 page allocation failure가 발생했을 때 swap out을 처리할 필요가 없습니다. 우선 그런 경우는 `PANIC ("todo")`로 표시하세요.

---

```
bool vm_do_claim_page (struct page *page);
```

> 페이지를 claim합니다. 여기서 claim이란 물리 프레임을 할당한다는 뜻입니다. 먼저 `vm_get_frame`을 호출해 frame을 얻습니다. 이 부분은 템플릿에 이미 작성되어 있습니다. 그런 다음 MMU를 설정해야 합니다. 즉 페이지 테이블에 가상 주소에서 물리 주소로 가는 mapping을 추가해야 합니다. 반환값은 작업이 성공했는지 여부를 나타내야 합니다.

---

```
bool vm_claim_page (void *va);
```

> `va`에 할당될 페이지를 claim합니다. 먼저 page를 얻은 뒤, 그 page로 `vm_do_claim_page`를 호출해야 합니다.
