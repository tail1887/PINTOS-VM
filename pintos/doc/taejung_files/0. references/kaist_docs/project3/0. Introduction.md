# Project3: Virtual Memory

지금쯤이면 Pintos의 내부 동작에 어느 정도 익숙해졌을 것입니다. 여러분의 OS는 적절한 동기화와 함께 여러 실행 thread를 올바르게 처리할 수 있고, 여러 사용자 프로그램을 동시에 로드할 수도 있습니다. 하지만 실행할 수 있는 프로그램의 수와 크기는 머신의 main memory 크기에 제한됩니다. 이번 과제에서는 무한한 메모리처럼 보이는 환상을 만들어 이 제한을 제거합니다.

이 과제는 이전 과제 위에 구현합니다. project 2의 테스트 프로그램도 project 3에서 동작해야 합니다. project 3 작업을 시작하기 전에 project 2 제출물의 버그를 고치도록 주의하세요. 그런 버그들은 project 3에서도 같은 문제를 일으킬 가능성이 높습니다.

project 3에서는 편의를 위해 단계별 지침을 제공합니다.

### 메모리 용어

먼저 메모리와 storage에 관한 몇 가지 용어를 소개합니다. 이 용어 중 일부는 project 2의 [Virtual Memory Layout](https://casys-kaist.github.io/pintos-kaist/project2/introduction.html)에서 익숙할 수 있지만, 새로 나오는 내용도 많습니다.

#### Pages

page, 때로 virtual page라고 부르는 것은 길이가 4,096바이트인 연속적인 가상 메모리 영역입니다. 이 크기를 **_page size_**라고 합니다. page는 반드시 **_page-aligned_**여야 합니다. 즉 page size로 나누어떨어지는 가상 주소에서 시작해야 합니다. 따라서 64비트 가상 주소의 마지막 12비트는 **_page offset_**, 또는 그냥 **_offset_**입니다. 상위 비트들은 곧 소개할 page table의 index를 나타내는 데 사용됩니다. 64비트 시스템에서는 4-level page table을 사용하므로 가상 주소는 다음과 같습니다.

```
63          48 47            39 38            30 29            21 20         12 11         0
+-------------+----------------+----------------+----------------+-------------+------------+
| Sign Extend |    Page-Map    | Page-Directory | Page-directory |  Page-Table |    Page    |
|             | Level-4 Offset |    Pointer     |     Offset     |   Offset    |   Offset   |
+-------------+----------------+----------------+----------------+-------------+------------+
              |                |                |                |             |            |
              +------- 9 ------+------- 9 ------+------- 9 ------+----- 9 -----+---- 12 ----+
                                          Virtual Address
```

각 프로세스는 독립적인 user virtual page 집합을 가집니다. 이는 가상 주소 `KERN_BASE`(0x8004000000) 아래에 있는 페이지들입니다. 반면 **_kernel virtual page_** 집합은 전역적이며, 어떤 thread나 process가 실행 중인지와 관계없이 같은 위치에 남아 있습니다. 커널은 user page와 kernel page 모두에 접근할 수 있지만, 사용자 프로세스는 자신의 user page에만 접근할 수 있습니다. 자세한 내용은 [Virtual Memory Layout](https://casys-kaist.github.io/pintos-kaist/project2/introduction.html)을 참고하세요.

Pintos는 가상 주소를 다루기 위한 여러 유용한 함수를 제공합니다. 자세한 내용은 [Virtual Addresses](https://casys-kaist.github.io/pintos-kaist/appendix/virtual_address.html) 섹션을 참고하세요.

#### Frames

**_frame_**, 때로 **_physical frame_** 또는 **_page frame_**이라고 부르는 것은 물리 메모리의 연속적인 영역입니다. page와 마찬가지로 frame도 page-size이고 page-aligned여야 합니다. 따라서 64비트 물리 주소는 다음처럼 **_frame number_**와 frame **_offset_**, 또는 그냥 **_offset_**으로 나눌 수 있습니다.

```
                          12 11         0
    +-----------------------+-----------+
    |      Frame Number     |   Offset  |
    +-----------------------+-----------+
              Physical Address
```

x86-64는 물리 주소의 메모리에 직접 접근하는 방법을 제공하지 않습니다. Pintos는 커널 가상 메모리를 물리 메모리에 직접 매핑하여 이를 우회합니다. 커널 가상 메모리의 첫 번째 page는 물리 메모리의 첫 번째 frame에, 두 번째 page는 두 번째 frame에 매핑되는 식입니다. 따라서 frame은 커널 가상 메모리를 통해 접근할 수 있습니다.

Pintos는 물리 주소와 커널 가상 주소 사이를 변환하는 함수를 제공합니다. 자세한 내용은 [Virtual Addresses](https://casys-kaist.github.io/pintos-kaist/appendix/virtual_address.html)를 참고하세요.

#### Page Tables

**_page table_**은 CPU가 가상 주소를 물리 주소로, 즉 page를 frame으로 변환하는 데 사용하는 자료구조입니다. page table 형식은 x86-64 아키텍처가 정합니다. Pintos는 `threads/mmu.c`에 page table management code를 제공합니다.

아래 그림은 page와 frame의 관계를 보여줍니다. 왼쪽의 가상 주소는 page number와 offset으로 구성됩니다. page table은 page number를 frame number로 변환하고, 변경되지 않은 offset과 결합하여 오른쪽의 물리 주소를 얻습니다.

```
                          +----------+
         .--------------->|Page Table|-----------.
        /                 +----------+            |
        |   12 11 0                               V  12 11 0
    +---------+----+                         +---------+----+
    | Page Nr | Ofs|                         |Frame Nr | Ofs|
    +---------+----+                         +---------+----+
     Virt Addr   |                            Phys Addr    ^
                  \_______________________________________/
```

#### Swap Slots

**_swap slot_**은 swap partition 안의 page-size 디스크 공간입니다. slot 배치를 지배하는 하드웨어 제한은 frame보다 더 유연하지만, page-align해서 나쁠 것이 없으므로 swap slot은 page-aligned여야 합니다.

### Resource Management 개요

다음 자료구조를 설계하고 구현해야 합니다.

**_Supplemental page table_**

> page table을 보완하여 page fault handling을 가능하게 합니다. 아래 Managing the Supplemental Page Table을 참고하세요.

**_Frame table_**

> physical frame의 eviction policy를 효율적으로 구현할 수 있게 합니다. 아래 Managing the Frame Table을 참고하세요.

**_Swap table_**

> swap slot의 사용 상태를 추적합니다. 아래 Managing the Swap Table을 참고하세요.

반드시 세 개의 완전히 별도인 자료구조를 구현할 필요는 없습니다. 관련 자원을 하나의 통합 자료구조에 전부 또는 일부 합치는 것이 편리할 수도 있습니다.

각 자료구조에 대해, 각 element가 어떤 정보를 담아야 하는지 결정해야 합니다. 또한 자료구조의 scope를 local, 즉 per-process로 할지, global, 즉 전체 시스템에 적용할지, 그리고 그 scope 안에서 몇 개의 instance가 필요한지도 결정해야 합니다.

설계를 단순화하기 위해, 이 자료구조들을 non-pageable memory에 저장해도 됩니다. 예를 들어 `calloc` 또는 `malloc`으로 할당한 메모리입니다. 이는 그들 사이의 pointer가 계속 유효하다고 확신할 수 있음을 뜻합니다.

#### 구현 선택지, Performance 관점

구현 선택지에는 array, list, bitmap, hash table 등이 있습니다. array는 보통 가장 단순한 접근이지만, 드문드문 채워진 array는 메모리를 낭비합니다. list도 단순하지만, 긴 list에서 특정 위치를 찾기 위해 순회하면 시간이 낭비됩니다. array와 list는 둘 다 크기를 조정할 수 있지만, list가 중간 삽입과 삭제를 더 효율적으로 지원합니다.

Pintos는 `lib/kernel/bitmap.c`와 `include/lib/kernel/bitmap.h`에 bitmap 자료구조를 포함합니다. bitmap은 bit의 배열이며, 각 bit는 true 또는 false가 될 수 있습니다. bitmap은 일반적으로 동일한 자원 집합의 사용 상태를 추적하는 데 사용됩니다. 자원 n이 사용 중이면 bitmap의 bit **_n_**이 true입니다. Pintos bitmap은 고정 크기이지만, 구현을 확장해 resizing을 지원할 수도 있습니다.

Pintos는 hash table 자료구조도 포함합니다. [Hash Table](https://casys-kaist.github.io/pintos-kaist/appendix/hash_table.html)을 참고하세요. Pintos hash table은 다양한 table size에서 insertion과 deletion을 효율적으로 지원합니다.

더 복잡한 자료구조는 더 나은 성능이나 다른 이점을 줄 수 있지만, 구현을 불필요하게 복잡하게 만들 수도 있습니다. 따라서 설계의 일부로 고급 자료구조, 예를 들어 balanced binary tree를 구현하는 것은 권장하지 않습니다.

### Supplemental Page Table 관리

**_supplemental_** page table은 각 페이지에 대한 추가 데이터로 page table을 보완합니다. page table 형식이 부과하는 제한 때문에 이것이 필요합니다. 이런 자료구조도 흔히 "page table"이라고 부르지만, 혼동을 줄이기 위해 "supplemental"이라는 단어를 붙입니다.

supplemental page table은 적어도 두 가지 목적에 사용됩니다. 가장 중요한 용도는 page fault가 발생했을 때, 커널이 fault가 난 virtual page를 supplemental page table에서 찾아 그곳에 어떤 데이터가 있어야 하는지 알아내는 것입니다. 두 번째로, 프로세스가 종료될 때 커널은 어떤 자원을 해제할지 결정하기 위해 supplemental page table을 참조합니다.

#### Supplemental Page Table의 구성

supplemental page table은 원하는 대로 구성해도 됩니다. 구성에 대한 기본 접근은 적어도 두 가지가 있습니다. segment 단위로 구성하거나 page 단위로 구성하는 방식입니다. 여기서 segment란 실행 파일 또는 memory-mapped file을 포함하는 메모리 영역, 즉 연속된 page 그룹을 말합니다.

선택 사항으로, page table 자체를 사용해 supplemental page table의 member를 추적할 수도 있습니다. 이를 위해서는 `threads/mmu.c`의 Pintos page table 구현을 수정해야 합니다. 이 접근은 고급 학생에게만 권장합니다.

#### Page fault 처리

supplemental page table의 가장 중요한 사용자는 page fault handler입니다. project 2에서는 page fault가 항상 커널 또는 사용자 프로그램의 버그를 의미했습니다. project 3에서는 더 이상 그렇지 않습니다. 이제 page fault는 단지 파일이나 swap slot에서 페이지를 가져와야 함을 의미할 수 있습니다. 이런 경우를 처리하기 위해 더 정교한 page fault handler를 구현해야 합니다. `userprog/exception.c`의 `page_fault()`는 `vm/vm.c`의 page fault handler인 `vm_try_handle_fault()`를 호출합니다. 여러분의 page fault handler는 대략 다음을 수행해야 합니다.

1. supplemental page table에서 fault가 난 page를 찾습니다. 메모리 참조가 유효하다면 supplemental page table entry를 사용해 페이지에 들어갈 데이터를 찾습니다. 그 데이터는 file system에 있거나, swap slot에 있거나, 단순히 all-zero page일 수 있습니다. sharing, 즉 Copy-on-Write를 구현했다면 페이지의 데이터가 이미 page table에는 없지만 page frame에는 있을 수도 있습니다. supplemental page table이 사용자 프로세스가 접근하려던 주소에 어떤 데이터도 기대해서는 안 된다고 나타내거나, 페이지가 kernel virtual memory 안에 있거나, 접근이 read-only page에 쓰려는 시도라면 그 접근은 invalid입니다. invalid access는 프로세스를 종료시키고 그 결과 모든 자원을 해제합니다.
    
2. 페이지를 저장할 frame을 얻습니다. sharing을 구현했다면 필요한 데이터가 이미 frame 안에 있을 수 있으며, 이 경우 해당 frame을 찾을 수 있어야 합니다.
    
3. file system이나 swap에서 읽거나, 0으로 채우는 등의 방식으로 데이터를 frame으로 가져옵니다. sharing을 구현했다면 필요한 페이지가 이미 frame 안에 있을 수 있으며, 이 단계에서 아무 작업도 필요하지 않을 수 있습니다.
    
4. fault가 난 가상 주소에 대한 page table entry가 물리 페이지를 가리키도록 합니다. `threads/mmu.c`의 함수들을 사용할 수 있습니다.
    

### Frame Table 관리

frame table은 각 frame마다 하나의 entry를 포함합니다. frame table의 각 entry는 현재 그 frame을 차지하고 있는 page가 있다면 그 page를 가리키는 pointer와, 여러분이 선택한 다른 데이터를 포함합니다. frame table을 통해 Pintos는 free frame이 없을 때 evict할 page를 선택하여 eviction policy를 효율적으로 구현할 수 있습니다.

user page에 사용되는 frame은 `palloc_get_page(PAL_USER)`를 호출해 "user pool"에서 얻어야 합니다. 일부 테스트 케이스가 예상치 못하게 실패하는 것을 피하려면 `PAL_USER`를 반드시 사용해야 합니다. frame table 구현의 일부로 `palloc.c`를 수정한다면 두 pool 사이의 구분을 유지해야 합니다.

frame table에서 가장 중요한 operation은 사용되지 않는 frame을 얻는 것입니다. free frame이 있으면 쉽습니다. free frame이 없으면 어떤 page를 frame에서 evict하여 frame을 free로 만들어야 합니다.

swap slot을 할당하지 않고는 evict할 수 있는 frame이 없는데 swap도 가득 찼다면, kernel을 panic시키세요. 실제 OS는 이런 상황에서 회복하거나 예방하기 위해 다양한 policy를 적용하지만, 이 project의 범위를 벗어납니다.

eviction 과정은 대략 다음 단계로 구성됩니다.

1. page replacement algorithm을 사용해 evict할 frame을 선택합니다. 아래에서 설명하는 page table의 "accessed" bit와 "dirty" bit가 유용할 것입니다.
    
2. 해당 frame을 참조하는 모든 page table에서 frame에 대한 참조를 제거합니다. sharing을 구현하지 않았다면 어떤 시점에도 하나의 page만 frame을 참조해야 합니다.
    
3. 필요하다면 page를 file system 또는 swap에 씁니다. evicted frame은 이후 다른 page를 저장하는 데 사용할 수 있습니다.
    

#### Accessed Bit와 Dirty Bit

x86-64 하드웨어는 각 page의 page table entry(PTE)에 있는 두 bit를 통해 page replacement algorithm 구현을 일부 도와줍니다. page를 읽거나 쓰면 CPU는 그 page의 PTE에서 accessed bit를 1로 설정하고, 쓰기가 발생하면 **_dirty bit_**를 1로 설정합니다. CPU는 이 bit들을 절대 0으로 재설정하지 않지만, OS는 그렇게 할 수 있습니다.

`aliases`, 즉 같은 frame을 참조하는 두 개 이상의 page를 인식해야 합니다. aliased frame에 접근하면 accessed bit와 dirty bit는 접근에 사용된 page에 대한 page table entry 하나에서만 갱신됩니다. 다른 alias들의 accessed bit와 dirty bit는 갱신되지 않습니다.

Pintos에서는 모든 user virtual page가 해당 kernel virtual page의 alias입니다. 이 alias들을 어떻게든 관리해야 합니다. 예를 들어 코드가 두 주소 모두에 대해 accessed bit와 dirty bit를 확인하고 갱신할 수 있습니다. 또는 커널이 user virtual address를 통해서만 user data에 접근하여 문제를 피할 수도 있습니다.

다른 alias는 sharing을 구현했거나 코드에 버그가 있는 경우에만 생겨야 합니다.

accessed bit와 dirty bit를 다루는 함수의 자세한 내용은 [Page Table Accessed and Dirty Bits](https://casys-kaist.github.io/pintos-kaist/appendix/page_table.html) 섹션을 참고하세요.

### Swap Table 관리

swap table은 사용 중인 swap slot과 free swap slot을 추적합니다. frame에서 page를 evict하여 swap partition으로 보낼 때 사용되지 않은 swap slot을 선택할 수 있어야 합니다. 또한 page가 다시 읽혀 오거나 swapped page를 가진 프로세스가 종료될 때 swap slot을 해제할 수 있어야 합니다.

`vm/build` 디렉터리에서 `pintos-mkdisk swap.dsk --swap-size=n` 명령을 사용해 n-MB swap partition을 포함하는 `swap.dsk`라는 디스크를 만드세요. 이후 pintos를 실행하면 `swap.dsk`가 자동으로 추가 디스크로 연결됩니다. 또는 `--swap-size=n`을 사용해 단일 실행에 대해 임시 n-MB swap disk를 사용하라고 pintos에 지시할 수 있습니다.

swap slot은 lazy하게 할당되어야 합니다. 즉 eviction에 실제로 필요할 때만 할당해야 합니다. 프로세스 시작 시점에 실행 파일에서 data page를 읽고 즉시 swap에 쓰는 것은 lazy가 아닙니다. 특정 페이지를 저장하기 위해 swap slot을 미리 예약해서도 안 됩니다.

swap slot의 내용이 frame으로 다시 읽히면 해당 swap slot을 해제하세요.

### Memory Mapped Files 관리

file system은 가장 흔하게 `read`와 `write` 시스템 콜로 접근됩니다. 보조 인터페이스로 `mmap` 시스템 콜을 사용해 파일을 가상 page에 "map"할 수 있습니다. 그러면 프로그램은 파일 데이터에 직접 memory instruction을 사용할 수 있습니다. 파일 `foo`의 길이가 `0x1000`바이트, 즉 4 kB 또는 한 페이지라고 가정해봅시다. `foo`가 주소 `0x5000`에서 시작하도록 메모리에 매핑되어 있다면, `0x5000...0x5fff` 위치에 대한 모든 메모리 접근은 `foo`의 대응하는 바이트에 접근합니다.

다음은 `mmap`을 사용해 파일을 콘솔에 출력하는 프로그램입니다. 명령줄에서 지정된 파일을 열고, 가상 주소 `0x10000000`에 매핑한 뒤, 매핑된 데이터를 콘솔(fd 1)에 쓰고, 파일을 unmap합니다.

```
#include <stdio.h>
#include <syscall.h>
int main (int argc UNUSED, char *argv[])
{
  void *data = (void *) 0x10000000;                 /* 매핑할 주소. */
  int fd = open (argv[1]);                          /* 파일 열기. */
  void *map = mmap (data, filesize (fd), 0, fd, 0); /* 파일 매핑. */
  write (1, data, filesize (fd));                   /* 파일을 콘솔에 쓰기. */
  munmap (map);                                     /* 파일 매핑 해제(선택 사항). */
  return 0;
}
```

제출물은 memory mapped file이 사용하는 메모리를 추적할 수 있어야 합니다. 이는 mapped region에서 발생하는 page fault를 올바르게 처리하고, mapped file이 프로세스 안의 다른 segment와 겹치지 않도록 보장하는 데 필요합니다.
