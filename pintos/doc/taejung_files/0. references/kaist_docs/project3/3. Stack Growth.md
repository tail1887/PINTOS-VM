프로젝트 2에서 스택은 `USER_STACK`에서 시작하는 단일 페이지였고, 프로그램 실행은 이 크기로 제한되었습니다. 이제 스택이 현재 크기를 넘어 커지면, 필요한 만큼 추가 페이지를 할당합니다.

추가 페이지는 스택 접근처럼 "보이는" 경우에만 할당하세요. 스택 접근과 다른 접근을 구분하려는 휴리스틱을 설계해야 합니다.

일반적인 실제 OS는 프로세스를 언제든 인터럽트하여 스택의 데이터를 수정하는 "signal"을 전달할 수 있으므로, 사용자 프로그램이 스택 포인터 아래쪽에 쓰기를 한다면 그 프로그램은 버그가 있는 것입니다. 하지만 x86-64의 PUSH 명령어는 스택 포인터를 조정하기 전에 접근 권한을 검사하므로, 스택 포인터보다 8바이트 아래에서 페이지 폴트를 일으킬 수 있습니다.

사용자 프로그램의 현재 스택 포인터 값을 얻을 수 있어야 합니다. 시스템 콜 안이나 사용자 프로그램이 발생시킨 페이지 폴트 안에서는 각각 `syscall_handler()` 또는 `page_fault()`에 전달된 `struct intr_frame`의 `rsp` 멤버에서 이를 가져올 수 있습니다. 잘못된 메모리 접근을 감지하기 위해 페이지 폴트에 의존한다면, 페이지 폴트가 커널에서 발생하는 또 다른 경우도 처리해야 합니다. 프로세서는 예외가 사용자 모드에서 커널 모드로 전환을 일으킬 때만 스택 포인터를 저장하므로, `page_fault()`에 전달된 `struct intr_frame`에서 `rsp`를 읽으면 사용자 스택 포인터가 아니라 정의되지 않은 값이 나올 수 있습니다. 따라서 사용자 모드에서 커널 모드로 처음 전환될 때 `rsp`를 `struct thread`에 저장하는 식의 다른 방법을 마련해야 합니다.

**스택 증가 기능을 구현하세요.** 이를 구현하려면 먼저 `vm/vm.c`의 `vm_try_handle_fault`를 수정해 스택 증가를 식별해야 합니다. 스택 증가임을 식별한 뒤에는 `vm/vm.c`의 `vm_stack_growth`를 호출하여 스택을 키워야 합니다. `vm_stack_growth`를 구현하세요.

---

```
bool vm_try_handle_fault (struct intr_frame *f, void *addr,
    bool user, bool write, bool not_present);
```

> 이 함수는 페이지 폴트 예외를 처리하는 동안 `userprog/exception.c`의 `page_fault`에서 호출됩니다. 이 함수 안에서 페이지 폴트가 스택 증가로 처리할 수 있는 유효한 경우인지 확인해야 합니다. 해당 폴트를 스택 증가로 처리할 수 있다고 판단했다면, 폴트가 난 주소를 인자로 `vm_stack_growth`를 호출하세요.

---

```
void vm_stack_growth (void *addr);
```

> `addr`이 더 이상 폴트가 나는 주소가 아니도록 하나 이상의 익명 페이지를 할당해 스택 크기를 늘립니다. 할당을 처리할 때 `addr`을 PGSIZE 단위로 내림 정렬해야 합니다.

대부분의 OS는 스택 크기에 절대적인 제한을 둡니다. 어떤 OS는 많은 Unix 시스템의 `ulimit` 명령처럼 사용자가 제한을 조정할 수 있게 합니다. 많은 GNU/Linux 시스템에서 기본 제한은 8 MB입니다. 이 프로젝트에서는 스택 크기를 최대 1 MB로 제한해야 합니다.

이제 모든 stack-growth 테스트 케이스가 통과해야 합니다.
