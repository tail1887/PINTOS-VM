**Pintos에 copy-on-write 메커니즘을 구현하세요.**

Copy-on-write는 동일한 물리 페이지 인스턴스를 사용해 복제 작업을 더 빠르게 만드는 자원 관리 기법입니다. 어떤 자원이 여러 프로세스에서 사용되고 있다면, 일반적으로 각 프로세스는 충돌이 생기지 않도록 그 자원의 사본을 가져야 합니다. 하지만 그 자원이 수정되지 않고 읽히기만 한다면, 물리 메모리에 여러 사본을 둘 필요가 없습니다.

예를 들어 `fork`를 통해 새 프로세스가 생성된다고 해봅시다. 자식은 부모의 자원을 자신의 가상 주소 공간으로 복제하여 상속해야 합니다. 보통 가상 메모리에 내용을 추가하려면 물리 페이지를 할당하고, 데이터를 프레임에 쓰고, 페이지 테이블에 virtual->physical 매핑을 추가해야 합니다. 이 단계들은 꽤 시간이 걸릴 수 있습니다.

하지만 copy-on-write 기법을 사용하면 자원의 새 사본을 위해 새로운 물리 페이지를 할당하지 않습니다. 기술적으로는 그 내용이 이미 물리 메모리에 존재하기 때문입니다. 따라서 자식 프로세스의 페이지 테이블에 virtual->physical 매핑만 추가합니다. 이때 가상 주소는 자식의 메모리 공간 안에 있습니다. 그러면 부모와 자식은 같은 물리 페이지 안의 같은 데이터에 접근합니다. 그러나 이들은 여전히 별도의 가상 주소 공간을 통해 격리되어 있으며, 같은 프레임을 참조하고 있다는 사실은 OS만 알고 있습니다. 공유 자원의 내용을 수정하려는 프로세스가 나타날 때만, 해당 프로세스를 위해 새로운 물리 페이지에 별도 사본을 만듭니다. 따라서 실제 복사 작업은 첫 번째 쓰기 시점까지 지연됩니다.

즉, OS는 copy-on-write 페이지에 대한 쓰기 시도를 감지할 수 있어야 합니다. 이를 위해 OS는 "write-protect" 메커니즘을 사용합니다. 아이디어는 단순합니다. 쓰기 접근에서 페이지 폴트를 발생시키는 것입니다. 메모리 관리 시스템의 지원을 이용하면 write-protected 페이지를 쓰기 불가능으로 표시하는 것만으로 쉽게 구현할 수 있습니다.

여러분은 **`fork`에 대해서만 copy-on-write를 구현하면 됩니다**. 자식 프로세스가 부모 프로세스의 자원을 상속할 때, 자식이 그것을 수정하려고 하기 전까지는 같은 물리 데이터를 참조할 수 있습니다. **모든 write-protected 페이지는 eviction 후보입니다.**

copy-on-write에 대해서는 기본 테스트 케이스만 제공합니다. 가능한 모든 경우를 고려해야 합니다. 작은 힌트를 하나 주자면, file-backed page의 공유도 구현해야 합니다. 이 extra project의 채점은 hidden test case로도 진행됩니다.
