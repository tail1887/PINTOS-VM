이 프로젝트의 이 부분에서는 _anonymous page_라고 부르는 디스크 기반이 아닌 이미지를 구현합니다.

Anonymous mapping에는 backing file이나 device가 없습니다. file-backed page와 달리 이름이 붙은 파일 소스가 없기 때문에 anonymous라고 부릅니다. Anonymous page는 실행 파일에서 스택과 힙 같은 용도로 사용됩니다.

익명 페이지를 설명하는 구조체로 `include/vm/anon.h`의 `anon_page`가 있습니다. 현재는 비어 있지만, 구현 과정에서 익명 페이지에 필요한 정보나 상태를 저장하기 위해 멤버를 추가할 수 있습니다. 또한 페이지의 일반 정보를 담고 있는 `include/vm/page.h`의 `struct page`도 살펴보세요. 익명 페이지의 경우 page 구조체 안에 `struct anon_page anon`이 포함됩니다.

### Lazy Loading을 이용한 Page Initialization

Lazy loading은 메모리 적재를 실제로 필요한 시점까지 지연하는 설계입니다. 페이지는 할당됩니다. 즉 해당 페이지에 대응하는 page struct는 존재합니다. 그러나 전용 물리 프레임은 없고, 페이지의 실제 내용도 아직 적재되지 않았습니다. 내용은 진짜로 필요해지는 시점에만 적재되며, 그 시점은 page fault로 신호를 받습니다.

페이지 타입이 세 가지이므로, 초기화 루틴도 각 페이지마다 다릅니다. 아래 섹션에서 다시 설명하겠지만, 여기서는 page initialization 흐름의 큰 그림을 제공합니다. 먼저 커널이 새 페이지 요청을 받으면 `vm_alloc_page_with_initializer`가 호출됩니다. initializer는 page 구조체를 할당하고 페이지 타입에 맞는 적절한 initializer를 설정하여 새 페이지를 초기화한 뒤, 제어를 사용자 프로그램으로 돌려줍니다. 사용자 프로그램이 실행되다가 어느 시점에, 자신이 가지고 있다고 믿는 페이지에 접근하려 하지만 그 페이지에 아직 내용이 없기 때문에 page fault가 발생합니다. fault handling 과정에서 `uninit_initialize`가 호출되고, 앞서 설정한 initializer를 호출합니다. 익명 페이지의 initializer는 `anon_initializer`이고, file-backed page의 initializer는 `file_backed_initializer`입니다.

페이지의 생명 주기는 initialize->(page_fault->lazy-load->swap-in>swap-out->...)->destroy 형태일 수 있습니다. 생명 주기의 각 전환마다 필요한 절차는 페이지 타입 또는 `VM_TYPE`에 따라 다르며, 앞 문단은 initialization에 대한 예시였습니다. 이 프로젝트에서는 각 페이지 타입에 대해 이러한 전환 과정을 구현합니다.

### Executable을 위한 Lazy Loading

Lazy loading에서는 프로세스가 실행을 시작할 때 즉시 필요한 메모리 부분만 메인 메모리에 적재됩니다. 이는 전체 binary image를 한 번에 메모리에 적재하는 eager loading에 비해 overhead를 줄일 수 있습니다.

Lazy loading을 지원하기 위해 `include/vm/vm.h`에 `VM_UNINIT`이라는 페이지 타입을 도입합니다. 모든 페이지는 처음에 `VM_UNINIT` 페이지로 생성됩니다. 또한 초기화되지 않은 페이지를 위한 page 구조체인 `include/vm/uninit.h`의 `struct uninit_page`도 제공합니다. 초기화되지 않은 페이지를 생성하고, 초기화하고, 파괴하는 함수들은 `include/vm/uninit.c`에서 찾을 수 있습니다. 이 함수들은 나중에 완성해야 합니다.

page fault가 발생하면 page fault handler(`userprog/exception.c`의 `page_fault`)는 `vm/vm.c`의 `vm_try_handle_fault`로 제어를 넘깁니다. 이 함수는 먼저 유효한 page fault인지 검사합니다. 여기서 _valid_란 잘못된 접근이 아닌 fault를 의미합니다. _bogus_ fault라면, 페이지에 어떤 내용을 적재하고 사용자 프로그램으로 제어를 돌려줍니다.

bogus page fault에는 세 가지 경우가 있습니다. lazy-loaded page, swapped-out page, write-protected page입니다. write-protected page는 [Copy-on-Write (Extra)](https://casys-kaist.github.io/pintos-kaist/project3/cow.html)를 참고하세요. 지금은 첫 번째 경우인 lazy-loaded page만 고려하세요. lazy loading을 위한 page fault라면, 커널은 segment를 lazy load하기 위해 `vm_alloc_page_with_initializer`에서 이전에 설정한 initializer 중 하나를 호출합니다. `userprog/process.c`의 `lazy_load_segment`를 구현해야 합니다.

**`vm_alloc_page_with_initializer()`를 구현하세요.** 전달된 vm_type에 따라 적절한 initializer를 가져오고, 그것을 사용해 `uninit_new`를 호출해야 합니다.

---

```
bool vm_alloc_page_with_initializer (enum vm_type type, void *va,
        bool writable, vm_initializer *init, void *aux);
```

> 주어진 type으로 uninitialized page를 생성합니다. uninit page의 swap_in handler는 type에 따라 자동으로 페이지를 초기화하고, 주어진 AUX로 INIT을 호출합니다. page struct를 얻으면 프로세스의 supplementary page table에 그 page를 삽입하세요. `vm.h`에 정의된 `VM_TYPE` 매크로가 유용할 수 있습니다.

page fault handler는 호출 체인을 따라가다가 swap_in을 호출할 때 최종적으로 `uninit_intialize`에 도달합니다. 이 함수의 완전한 구현은 제공됩니다. 다만 설계에 따라 `uninit_initialize`를 수정해야 할 수도 있습니다.

---

```
static bool uninit_initialize (struct page *page, void *kva);
```

> 첫 번째 fault에서 페이지를 초기화합니다. 템플릿 코드는 먼저 `vm_initializer`와 `aux`를 가져오고, 함수 포인터를 통해 대응하는 page_initializer를 호출합니다. 설계에 따라 이 함수를 수정해야 할 수도 있습니다.

필요에 따라 `vm/anon.c`의 `vm_anon_init`과 `anon_initializer`를 수정할 수 있습니다.

---

```
void vm_anon_init (void);
```

> anonymous page subsystem을 초기화합니다. 이 함수 안에서 anonymous page와 관련된 모든 것을 설정할 수 있습니다.

---

```
bool anon_initializer (struct page *page,enum vm_type type, void *kva);
```

> 이 함수는 먼저 `page->operations`에 anonymous page용 handler를 설정합니다. 현재 비어 있는 구조체인 `anon_page`의 일부 정보를 갱신해야 할 수도 있습니다. 이 함수는 anonymous page, 즉 `VM_ANON`의 initializer로 사용됩니다.

**`userprog/process.c`의 `load_segment`와 `lazy_load_segment`를 구현하세요.** 실행 파일에서 segment를 적재하는 기능을 구현합니다. 이 모든 페이지는 lazy하게 적재되어야 합니다. 즉 커널이 해당 페이지의 page fault를 가로챌 때만 적재되어야 합니다.

프로그램 loader의 핵심인 `userprog/process.c`의 `load_segment` 루프를 수정해야 합니다. 루프를 돌 때마다 `vm_alloc_page_with_initializer`를 호출해 pending page object를 생성합니다. page fault가 발생하는 시점이 바로 segment가 실제로 파일에서 적재되는 시점입니다.

---

```
static bool load_segment (struct file *file, off_t ofs, uint8_t *upage,
        uint32_t read_bytes, uint32_t zero_bytes, bool writable);
```

> 현재 코드는 main loop 안에서 파일에서 읽을 바이트 수와 0으로 채울 바이트 수를 계산합니다. 그런 다음 `vm_alloc_page_with_initializer`를 호출해 pending object를 생성합니다. `vm_alloc_page_with_initializer`에 `aux` 인자로 넘길 auxiliary value를 설정해야 합니다. binary loading에 필요한 정보를 담는 구조체를 만들고 싶을 수 있습니다.

---

```
static bool lazy_load_segment (struct page *page, void *aux);
```

> `load_segment`에서 `vm_alloc_page_with_initializer`의 네 번째 인자로 `lazy_load_segment`가 전달되는 것을 보았을 것입니다. 이 함수는 executable page의 initializer이며 page fault 시점에 호출됩니다. page struct와 `aux`를 인자로 받습니다. `aux`는 `load_segment`에서 설정한 정보입니다. 이 정보를 사용해 segment를 읽을 파일을 찾고, 최종적으로 segment를 메모리에 읽어와야 합니다.

**새 메모리 관리 시스템에 맞게 `userprog/process.c`의 `setup_stack`을 조정해야 합니다.** 첫 번째 stack page는 lazy하게 할당할 필요가 없습니다. load time에 command line argument와 함께 할당하고 초기화할 수 있으며, fault가 날 때까지 기다릴 필요가 없습니다. stack을 식별하는 방법을 제공해야 할 수도 있습니다. `vm/vm.h`의 `vm_type`에 있는 auxiliary marker, 예를 들어 `VM_MARKER_0`을 사용해 페이지를 표시할 수 있습니다.

마지막으로, `vm_try_handle_fault` 함수를 수정해 supplemental page table을 `spt_find_page`로 조회하여 fault가 난 주소에 대응하는 page struct를 찾도록 하세요.

모든 요구사항을 구현한 뒤에는 fork를 제외한 project 2의 모든 테스트가 통과해야 합니다.

### Supplemental Page Table - 다시 보기

이제 copy와 cleanup operation을 지원하기 위해 supplemental page table interface를 다시 살펴봅니다. 이 operation들은 프로세스를 생성할 때, 더 구체적으로는 자식 프로세스를 만들 때, 또는 프로세스를 파괴할 때 필요합니다. 자세한 내용은 아래에서 설명합니다. 지금 supplemental page table을 다시 보는 이유는 위에서 구현한 initializing function 일부를 사용하고 싶을 수 있기 때문입니다.

**`vm/vm.c`에 `supplemental_page_table_copy`와 `supplemental_page_table_kill`을 구현하세요.**

---

```
bool supplemental_page_table_copy (struct supplemental_page_table *dst,
    struct supplemental_page_table *src);
```

> supplemental page table을 src에서 dst로 복사합니다. 이는 자식이 부모의 실행 context를 상속해야 할 때, 즉 `fork()`에서 사용됩니다. src의 supplemental page table 안의 각 page를 순회하며, 해당 entry의 정확한 복사본을 dst의 supplemental page table에 만드세요. uninit page를 할당하고 즉시 claim해야 합니다.

---

```
void supplemental_page_table_kill (struct supplemental_page_table *spt);
```

> supplemental page table이 보유하고 있던 모든 자원을 해제합니다. 이 함수는 프로세스가 종료될 때(`userprog/process.c`의 `process_exit()`) 호출됩니다. table의 page entry들을 순회하며 table 안의 page들에 대해 `destroy(page)`를 호출해야 합니다. 이 함수 안에서는 실제 page table(pml4)과 물리 메모리(palloc된 메모리)를 걱정할 필요가 없습니다. supplemental page table이 정리된 뒤 호출자가 그것들을 정리합니다.

### Page Cleanup

**`vm/uninit.c`의 `uninit_destroy`와 `vm/anon.c`의 `anon_destroy`를 구현하세요.** 이것은 uninitialized page에 대한 `destroy` operation의 handler입니다. uninitialized page는 다른 page object로 변형되지만, 프로세스가 종료될 때 여전히 uninit page가 남아 있을 수 있습니다.

---

```
static void uninit_destroy (struct page *page);
```

> page struct가 보유하던 자원을 해제합니다. page의 vm type을 확인하고 그에 맞게 처리하고 싶을 수 있습니다.

지금은 anonymous page만 처리하면 됩니다. 나중에 file-backed page를 정리하기 위해 이 함수를 다시 살펴보게 됩니다.

---

```
static void anon_destroy (struct page *page);
```

> anonymous page가 보유하던 자원을 해제합니다. 명시적으로 page struct를 free할 필요는 없습니다. 호출자가 처리해야 합니다.
