# System Calls

원문: [System Calls](https://casys-kaist.github.io/pintos-kaist/project2/system_call.html)

## System Calls

**시스템 콜 인프라를 구현합니다.**

`userprog/syscall.c`에 시스템 콜 핸들러를 구현하세요. 제공되는 골격 구현은 시스템 콜이 오면 프로세스를 종료시키는 식으로만 동작합니다. 시스템 콜 번호를 꺼내고, 필요한 인자들을 읽은 뒤, 적절한 동작을 수행하도록 바꿔야 합니다.

### System Call Details

첫 번째 프로젝트에서 이미 다룬 것처럼, OS가 사용자 프로그램으로부터 제어를 되찾는 한 가지 방법은 타이머·I/O 장치 인터럽트입니다. 이것들은 CPU 바깥에서 발생하므로 "외부" 인터럽트입니다.

OS는 또한 프로그램 코드에서 일어나는 **소프트웨어 예외**(페이지 폴트, 0으로 나누기 등)를 처리합니다. 예외는 사용자 프로그램이 OS에 서비스("시스템 콜")를 요청하는 수단이기도 합니다.

전통적인 x86에서는 시스템 콜이 다른 소프트웨어 예외와 같은 방식으로 처리되었습니다. 하지만 x86-64에서는 시스템 콜 전용 명령 `syscall`이 도입되어 핸들러 호출이 빨라집니다.

오늘날 x86-64에서 시스템 콜을 거는 가장 흔한 방법은 `syscall` 명령입니다. Pintos에서도 사용자 프로그램은 `syscall`로 시스템 콜을 합니다. 시스템 콜 번호와 추가 인자는 `syscall`을 실행하기 전에 보통의 방식으로 레지스터에 넣되, **다음 두 가지만 예외**입니다.

- `%rax`에는 시스템 콜 번호가 들어갑니다.
- 네 번째 인자는 `%rcx`가 아니라 **`%r10`** 입니다.

따라서 시스템 콜 핸들러 `syscall_handler()`가 제어를 받을 때, 시스템 콜 번호는 rax에 있고, 인자는 `%rdi`, `%rsi`, `%rdx`, `%r10`, `%r8`, `%r9` 순입니다.

호출자의 레지스터는 넘겨받은 `struct intr_frame`으로 접근할 수 있습니다(`struct intr_frame`은 커널 스택 위에 있습니다).

x86-64에서 함수 반환값은 RAX에 둡니다. 값을 반환하는 시스템 콜은 `struct intr_frame`의 `rax` 멤버를 수정해 반환하면 됩니다.

### 다음 시스템 콜을 구현하세요

아래 프로토타입은 사용자 프로그램이 `include/lib/user/syscall.h`를 포함할 때 보는 것과 같습니다(`include/lib/user` 아래 헤더는 **사용자 프로그램 전용**). 각 시스템 콜 번호는 `include/lib/syscall-nr.h`에 정의되어 있습니다.

---

```c
void halt (void);
```

> `power_off()`를 호출해 Pintos를 종료합니다(`src/include/threads/init.h` 선언). 가능한 한 드물게 쓰지 마세요. 교착 가능성 등 유용한 정보를 잃기 쉽습니다.

---

```c
void exit (int status);
```

> 현재 사용자 프로그램을 종료하고 `status`를 커널에 돌려줍니다. 부모가 `wait`로 기다리면(아래 참고) 이 값이 반환됩니다. 관례적으로 `status`가 `0`이면 성공, 0이 아니면 오류입니다.

---

```c
pid_t fork (const char *thread_name);
```

> 현재 프로세스의 복제인 새 프로세스를 `THREAD_NAME` 이름으로 만듭니다. **캘리 저장(callee-saved) 레지스터**인 `%RBX`, `%RSP`, `%RBP`, `%R12`–`%R15`를 제외한 레지스터 값은 복제할 필요가 없습니다. 자식의 pid를 반환해야 하며, 그렇지 않으면 유효한 pid가 되면 안 됩니다. 자식 프로세스에서는 반환값이 `0`이어야 합니다. 자식은 파일 디스크립터·가상 메모리 공간을 포함한 자원이 **복제(DUPLICATE)**되어야 합니다. 부모는 자식이 성공적으로 복제되었는지 알기 전에는 **`fork`에서 돌아오면 안 됩니다.** 즉, 자식이 자원 복제에 실패하면 부모의 `fork()` 호출은 `TID_ERROR`를 반환해야 합니다.

> 템플릿은 `threads/mmu.c`의 `pml4_for_each()`를 사용해 페이지 테이블 구조를 포함한 전체 사용자 메모리 공간을 복사하지만, 넘겨진 `pte_for_each_func`의 빈 부분은 여러분이 채워야 합니다([가상 주소](https://casys-kaist.github.io/pintos-kaist/appendix/virtual_address.html) 참고).

---

```c
int exec (const char *cmd_line);
```

> `cmd_line`에 주어진 실행 파일로 **현재 프로세스를 바꿉니다.** 인자도 함께 넘깁니다. 성공하면 **절대 돌아오지 않습니다.** 어떤 이유로든 프로그램을 적재·실행할 수 없으면 프로세스는 exit 상태 `-1`로 **종료**됩니다. `exec`를 호출한 스레드의 이름은 바꾸지 않습니다. **`exec` 호출 후에도 파일 디스크립터는 열린 채로 유지됩니다.**

---

```c
int wait (pid_t pid);
```

> 자식 프로세스 `pid`가 끝날 때까지 기다리고 종료 상태를 가져옵니다. `pid`가 아직 살아 있으면 종료될 때까지 블록합니다. 그다음 `pid`가 `exit`에 넘긴 상태를 반환합니다. `pid`가 `exit()`를 호출하지 않았지만 커널에 의해 종료된 경우(예: 예외로 kill) `wait(pid)`는 반드시 `-1`을 반환해야 합니다. 부모가 `wait`를 호출할 때 이미 자식이 종료된 뒤여도 기다리는 것은 합법이며, 커널은 그래도 부모가 자식의 exit 상태를 가져가거나, 자식이 커널에 의해 종료되었음을 알 수 있게 해야 합니다.

> 다음 중 하나라도 참이면 `wait`는 즉시 실패하고 `-1`을 반환해야 합니다.

> - `pid`가 호출 프로세스의 **직접 자식**이 아닌 경우. `pid`는 성공한 `fork`의 반환값으로만 직접 자식입니다. 자식은 **승계되지 않습니다**: A가 B를, B가 C를 낳았다면 B가 죽었어도 A는 C를 기다릴 수 없습니다. A의 `wait(C)`는 실패해야 합니다. 마찬가지로 부모가 먼저 exit해도 고아 프로세스가 새 부모에게 붙지는 않습니다.
> - 이미 그 `pid`에 대해 `wait`를 한 번 호출한 경우. 즉, 주어진 자식에 대해 `wait`는 **최대 한 번**입니다.

> 프로세스는 자식을 여러 명 낳을 수 있고, 아무 순서로 `wait`할 수 있으며, 일부·전부 자식을 기다리지 않고 exit할 수도 있습니다. 설계 시 `wait`가 일어날 수 있는 모든 경우를 고려하세요. 부모가 기다리든 말든, 자식이 부모보다 먼저·나중에 죽든, 프로세스의 모든 자원(`struct thread` 포함)은 반드시 해제되어야 합니다.

> **Pintos는 최초 프로세스가 exit할 때까지 종료되면 안 됩니다.** 제공 코드는 `threads/init.c`의 `main()`에서 `userprog/process.c`의 `process_wait()`를 호출해 이를 맞추려 합니다. 함수 상단 주석에 따라 `process_wait()`를 구현한 뒤, `wait` 시스템 콜은 `process_wait()` 위에 올리는 것을 권장합니다.

> 이 시스템 콜은 나머지보다 구현 부담이 훨씬 큽니다.

---

```c
bool create (const char *file, unsigned initial_size);
```

> 이름이 `file`인 새 파일을 만들고 크기를 처음에 `initial_size`바이트로 둡니다. 성공하면 true, 아니면 false. 새 파일을 만든다고 **열리는 것은 아닙니다.** 열려면 별도의 `open` 시스템 콜이 필요합니다.

---

```c
bool remove (const char *file);
```

> 이름이 `file`인 파일을 삭제합니다. 성공하면 true, 아니면 false. 열려 있든 닫혀 있든 삭제할 수 있고, 열린 파일을 삭제한다고 닫히지는 않습니다. 자세한 내용은 [FAQ](https://casys-kaist.github.io/pintos-kaist/project2/FAQ.html)의 “Removing an Open File”을 보세요.

---

```c
int open (const char *file);
```

> 이름이 `file`인 파일을 엽니다. 성공하면 0 이상의 정수 핸들 “파일 디스크립터(fd)”를 반환하고, 열 수 없으면 `-1`. fd 0과 1은 콘솔용으로 예약되어 있습니다: fd 0(`STDIN_FILENO`)은 표준 입력, fd 1(`STDOUT_FILENO`)은 표준 출력입니다. `open` 시스템 콜은 **이 둘 중 하나를 반환하지 않습니다.** 다만 아래에서 명시한 대로 시스템 콜 인자로는 사용될 수 있습니다. 프로세스마다 fd 집합은 독립적이며, 자식은 부모의 fd를 상속합니다. 한 파일을 한 프로세스나 여러 프로세스에서 여러 번 열면 **매번 새 fd**가 나옵니다. 같은 파일에 대한 서로 다른 fd는 `close`가 따로따로 적용되며 **파일 위치를 공유하지 않습니다.** **추가(extra) 과제**를 할 때는 리눅스처럼 **0부터 증가하는 정수**를 쓰는 방식을 따르세요.

---

```c
int filesize (int fd);
```

> `fd`로 연 파일의 크기(바이트)를 반환합니다.

---

```c
int read (int fd, void *buffer, unsigned size);
```

> `fd`로 연 파일에서 `size`바이트를 `buffer`로 읽습니다. 실제로 읽은 바이트 수를 반환합니다(파일 끝이면 `0`). 파일 끝이 아닌 이유로 읽을 수 없으면 `-1`. fd 0은 `input_getc()`로 키보드에서 읽습니다.

---

```c
int write (int fd, const void *buffer, unsigned size);
```

> `buffer`에서 `size`바이트를 `fd`로 연 파일에 씁니다. 실제로 쓴 바이트 수를 반환하며, 일부만 쓸 수 있으면 그보다 작을 수 있습니다. 일반적으로는 파일 끝을 넘어 쓰면 파일이 늘어나지만, 기본 파일 시스템은 파일 성장을 구현하지 않았습니다. 기대 동작은 파일 끝까지 가능한 만큼 쓰고 실제로 쓴 수를 반환하거나, 한 바이트도 못 쓰면 `0`입니다. fd 1은 콘솔에 씁니다. 콘솔 출력 코드는 **가능하면 `buffer` 전체를 한 번의 `putbuf()` 호출로** 쓰세요(`size`가 수백 바이트 이하일 때). 더 크면 나누는 것도 합리적입니다. 그렇지 않으면 여러 프로세스의 출력 줄이 섞여 사람과 채점 스크립트를 혼란스럽게 합니다.

---

```c
void seek (int fd, unsigned position);
```

> `fd`로 연 파일에서 다음에 읽거나 쓸 바이트를 파일 시작부터 `position`바이트 위치로 옮깁니다(`position`이 `0`이면 파일 시작). 현재 파일 끝을 넘어가도 오류가 아닙니다. 이후 `read`는 0바이트를 읽어 EOF를 나타냅니다. 이후 `write`는 파일을 늘리고, 쓰이지 않은 구간은 0으로 채웁니다. (다만 Pintos는 프로젝트 4가 끝나기 전까지 파일 길이가 고정이므로, 파일 끝을 넘는 `write`는 오류를 반환합니다.) 이 의미는 파일 시스템에 구현되어 있으며 시스템 콜 쪽에서 특별히 할 일은 없습니다.

---

```c
unsigned tell (int fd);
```

> `fd`로 연 파일에서 다음에 읽거나 쓸 위치를 파일 시작부터 바이트 단위로 반환합니다.

---

```c
void close (int fd);
```

> fd `fd`를 닫습니다. 프로세스가 exit하거나 종료되면 **열려 있던 모든 fd가 암시적으로** 닫힌 것과 같습니다.

헤더에는 다른 시스템 콜도 정의되어 있습니다. 지금은 무시하세요. 프로젝트 3에서 일부, 프로젝트 4에서 나머지를 구현하므로 **확장 가능하게** 설계하세요.

**시스템 콜은 동기화해야 합니다.** 여러 사용자 프로세스가 동시에 시스템 콜을 호출할 수 있어야 합니다. 특히 `filesys` 디렉터리의 파일 시스템 코드는 **여러 스레드가 동시에 호출하면 안전하지 않습니다.** 시스템 콜 구현에서 파일 시스템 코드는 **임계 구역**으로 다루세요. `process_exec()`도 파일에 접근한다는 점을 잊지 마세요. 당분간 `filesys` 코드 수정은 권장하지 않습니다.

각 시스템 콜에 대한 사용자 수준 함수는 `lib/user/syscall.c`에 있습니다. C 프로그램에서 인라인 어셈블리로 시스템 콜을 걸고, 필요하면 반환값을 돌려줍니다.

이 부분을 끝낸 뒤로는 Pintos가 **탄탄해야 합니다.** 사용자 프로그램이 OS를 크래시·패닉·assert 실패·기타 오동작에 이르게 해서는 안 됩니다. 테스트는 시스템 콜을 아주 다양한 방식으로 깨뜨리려 할 것입니다. 모든 모서리 경우를 생각해 처리하세요. 사용자 프로그램이 OS를 멈출 수 있는 유일한 방법은 **halt 시스템 콜**이어야 합니다.

시스템 콜에 잘못된 인자가 넘어오면, (값을 반환하는 호출은) 오류 값 반환, 정의되지 않은 값 반환, 프로세스 종료 중 허용되는 방식으로 처리하면 됩니다.
