## FAQ

### 얼마나 많은 코드를 써야 하나요?

참고 해법에 대해 `git diff --stat`으로 요약하면 아래와 같습니다. 마지막 줄은 삽입·삭제된 총 줄 수이며, 바뀐 한 줄은 삽입과 삭제 모두에 잡힙니다. 참고 해법은 가능한 해법 중 하나일 뿐이며, 다른 훌륭한 해법도 많고 참고 해법과 크게 다를 수 있습니다. 참고 해법이 수정한 파일을 고치지 않은 해법도 있을 수 있고, 참고 해법이 건드리지 않은 파일을 수정한 해법도 있을 수 있습니다. 추가 요구사항 구현도 포함되어 있으며, 추가와 관련된 줄은 대략 150줄 정도입니다.

```
src/include/threads/thread.h   |  23 ++
src/include/userprog/syscall.h |   3 +
src/threads/thread.c           |   5 +
src/userprog/exception.c       |   4 +
src/userprog/process.c         | 355 +++++++++++++++++++++++++++++++++++++++++-
src/userprog/syscall.c         | 429 ++++++++++++++++++++++++++++++++++++++++++-
6 files changed, 782 insertions(+), 37 deletions(-)
```

### `pintos -p file -- -q`를 실행하면 커널이 항상 패닉합니다.

파일 시스템을 포맷했나요(`pintos -f`)?

파일 이름이 너무 길지 않나요? 파일 시스템은 파일 이름을 14자로 제한합니다.

`pintos -p ../../examples/echo -- -q` 같은 명령은 그 한도를 넘깁니다.

`pintos -p ../../examples/echo:echo -- -q`처럼 Pintos 쪽 이름을 `echo`로 줄여 넣으세요.

파일 시스템이 가득 찼나요?

이미 파일이 16개 있나요? 기본 Pintos 파일 시스템은 파일 16개 제한이 있습니다.

파일 시스템이 너무 조각나서 연속된 빈 공간이 부족할 수도 있습니다.

### `pintos -p ../file --`를 실행했는데 `file`이 복사되지 않습니다.

기본적으로는 **참조한 경로 문자열 그대로** 이름이 붙습니다. 이 경우 복사된 이름은 `'../file'`일 수 있습니다. 대신 `pintos -p ../file:file --`처럼 쓰세요.

### 사용자 프로그램이 전부 페이지 폴트로 죽습니다.

인자 전달을 구현하지 않았거나 잘못 구현했을 때 그렇습니다. 사용자용 기본 C 라이브러리는 스택에서 `argv`를 읽으려 합니다. 스택이나 레지스터가 맞게 설정되지 않으면 페이지 폴트가 납니다.

### 사용자 프로그램이 전부 `system call!`로 죽습니다.

시스템 콜을 구현해야 다른 동작을 볼 수 있습니다. 그럴듯한 프로그램은 최소 한 번은 시스템 콜을 하고, 대부분은 그보다 많이 합니다. 특히 `printf()`는 `write` 시스템 콜을 호출합니다. 기본 시스템 콜 핸들러는 `system call!`만 출력하고 프로그램을 끝냅니다. 그전까지는 인자 전달이 맞는지 `hex_dump()`로 확인할 수 있습니다.

### 사용자 프로그램을 디스어셈블하려면?

`objdump` 유틸리티로 사용자 프로그램 전체나 오브젝트 파일을 디스어셈블할 수 있습니다. `objdump -d file` 형태로 실행합니다. GDB의 `disassemble` 명령으로 개별 함수만 디스어셈블할 수도 있습니다.

### 왜 많은 C 헤더가 Pintos 프로그램에서 동작하지 않나요?

### Pintos 프로그램에 libfoo를 쓸 수 있나요?

제공하는 C 라이브러리는 매우 제한적입니다. 실제 OS의 C 라이브러리에서 기대하는 많은 기능이 없습니다. C 라이브러리는 OS(와 아키텍처)에 맞게 빌드되어야 하며, I/O와 메모리 할당 등은 시스템 콜을 써야 합니다(모든 함수가 그런 것은 아니지만, 보통 라이브러리는 한 덩어리로 컴파일됩니다).

원하는 라이브러리가 Pintos가 구현하지 않은 C 라이브러리 부분을 쓰고 있을 가능성이 큽니다. Pintos에서 동작하게 하려면 이식 작업이 적어도 어느 정도 필요합니다. 특히 Pintos 사용자 프로그램용 C 라이브러리에는 **`malloc()` 구현이 없습니다.**

### 새 사용자 프로그램은 어떻게 컴파일하나요?

`src/examples/Makefile`을 수정한 뒤 `make`를 실행합니다.

### 디버거로 사용자 프로그램을 실행할 수 있나요?

가능합니다. 다만 제약이 있습니다. GDB 문서를 보세요.

### `tid_t`와 `pid_t`의 차이는?

`tid_t`는 커널 스레드를 식별합니다. `process_fork()`로 만들었을 수도 있고(그 안에 사용자 프로세스가 있을 수 있음), `thread_create()`로 만들었을 수도 있습니다(사용자 프로세스 없음). 커널 안에서만 쓰는 타입입니다. `pid_t`는 사용자 프로세스를 식별하며, `exec`와 `wait` 시스템 콜에서 사용자와 커널이 함께 씁니다. `tid_t`와 `pid_t`에 어떤 타입을 쓸지는 여러분이 정하면 됩니다. 기본적으로 둘 다 `int`입니다. 같은 프로세스를 같은 값으로 매핑해도 되고, 더 복잡한 매핑을 써도 됩니다.

### `struct file *`를 캐스팅해서 파일 디스크립터를 얻으면 안 되나요?

### `struct thread *`를 `pid_t`로 캐스팅하면 안 되나요?

아니요. `pid_t`와 파일 디스크립터는 포인터 타입보다 작으므로 **직접 캐스팅할 수 있습니다.**

### 프로세스당 열 수 있는 파일 수에 상한을 둬도 되나요?

임의로 작은 상한은 두지 않는 편이 낫습니다. 필요하면 프로세스당 **최대 128개** 정도는 허용할 수 있습니다. 다만 **추가 요구사항**을 구현하려면 상한이 없어야 합니다.

### 열린 파일이 삭제되면 어떻게 되나요?

표준 Unix 파일 의미를 구현해야 합니다. 즉, 파일이 삭제된 뒤에도 그 파일을 가리키는 파일 디스크립터를 가진 프로세스는 **그 디스크립터로 계속 읽고 쓸 수 있습니다.** 파일 이름은 없어지고 다른 프로세스는 열 수 없지만, 모든 디스크립터가 닫히거나 머신이 꺼질 때까지 파일은 존재합니다.

### 스택 공간이 4kB보다 많이 필요한 사용자 프로그램은?

스택 설정 코드를 바꿔 프로세스마다 **한 페이지보다 큰 스택**을 할당할 수 있습니다. 다음 프로젝트에서 더 나은 해법을 구현합니다.
