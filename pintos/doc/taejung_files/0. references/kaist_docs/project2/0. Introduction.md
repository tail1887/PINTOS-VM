# Project2: User Programs

이제 Pintos에 익숙해지고 인프라와 스레드 패키지에 대한 감을 잡았으니, 사용자 프로그램을 실행할 수 있게 만드는 부분으로 넘어갈 차례입니다. 기본 코드는 이미 사용자 프로그램을 적재하고 실행하는 것까지 지원하지만, I/O나 상호작용은 아직 불가능합니다. 이번 프로젝트에서는 프로그램이 시스템 콜을 통해 OS와 상호작용할 수 있게 만듭니다. 과제 작업은 주로 `userprog` 디렉터리에서 진행하지만, Pintos의 거의 모든 다른 부분과도 맞닿게 됩니다. 관련 내용은 아래에서 설명합니다.

**프로젝트 2는 반드시 프로젝트 1 제출물 위에 구축해야 합니다.** 프로젝트 1 코드가 프로젝트 2 코드에 직접 영향을 주지는 않더라도, 증분 과제이므로 프로젝트 1 테스트도 계속 통과해야 합니다.

또한 프로젝트 2에는 **추가(extra) 과제**가 있습니다. 이는 **선택 구현**입니다. 추가 과제에는 테스트 코드를 제외하면 골격 코드가 거의 없으며, 설계는 전적으로 여러분에게 달려 있습니다. **추가 요구사항을 제출하고 테스트하려면 `userprog/Make.vars`를 수정해야 합니다.**

마지막으로, `TODO`가 없는 코드라고 해서 **반드시 수정할 필요가 없다는 뜻은 아닙니다.** 프로젝트 2에서는 테스트 코드를 제외한 소스 코드는 자유롭게 수정할 수 있습니다.

## Background

지금까지 Pintos 아래에서 실행한 코드는 모두 운영체제 커널의 일부였습니다. 예를 들어 지난 과제의 테스트 코드는 커널의 일부로 실행되어 시스템의 특권 영역에 완전히 접근할 수 있었습니다. 이제 OS 위에서 사용자 프로그램을 실행하기 시작하면 상황이 달라집니다. 이번 프로젝트는 그에 따른 결과를 다룹니다.

동시에 둘 이상의 프로세스가 실행될 수 있습니다. 각 프로세스는 스레드 하나만 가집니다(멀티스레드 프로세스는 지원하지 않음). 사용자 프로그램은 **전체 머신을 독점한다는 환상** 아래 작성됩니다. 따라서 여러 프로세스를 동시에 적재·실행할 때는 메모리, 스케줄링, 기타 상태를 올바르게 관리해 이 환상을 유지해야 합니다.

이전 프로젝트에서는 테스트 코드를 커널에 직접 컴파일해 넣었기 때문에, 커널 내부에 특정 함수 인터페이스를 요구할 수밖에 없었습니다. 앞으로는 **사용자 프로그램을 실행해** OS를 검증합니다. 그만큼 자유도가 커집니다. 사용자 프로그램 인터페이스가 여기서 설명하는 명세를 만족하기만 하면, 그 제약 안에서는 커널 코드를 마음대로 재구성하거나 다시 작성해도 됩니다.

다만 **여러분이 작성한 코드는 `#ifdef VM`으로 둘러싼 블록 안에 두면 안 됩니다.** 해당 블록은 프로젝트 3에서 가상 메모리 서브시스템을 켠 뒤에 포함됩니다. 반대로 `#ifndef VM`으로 둘러싼 코드는 프로젝트 3에서는 빠집니다.

시작하기 전에 [동기화(Synchronization)](https://casys-kaist.github.io/pintos-kaist/appendix/synchronization.html)와 [가상 주소(Virtual Address)](https://casys-kaist.github.io/pintos-kaist/appendix/virtual_address.html)를 읽어보는 것을 **강력히 권장**합니다.

### Source Files

무엇을 프로그래밍하게 될지 한눈에 보려면, 작업하게 될 부분을 차례로 훑는 것이 가장 쉽습니다. `userprog`에는 파일 수가 많지 않지만, **대부분의 실질 작업은 여기서 이뤄집니다.**

#### `process.c`, `process.h`

ELF 바이너리를 적재하고 프로세스를 시작합니다.

#### `syscall.c`, `syscall.h`

사용자 프로세스가 커널 기능을 쓰려면 시스템 콜을 호출합니다. 여기는 시스템 콜 핸들러의 골격입니다. 현재는 메시지를 출력하고 사용자 프로세스를 종료할 뿐입니다. 이번 프로젝트 2부에서는 시스템 콜에 필요한 나머지 동작을 모두 추가합니다.

#### `syscall-entry.S`

시스템 콜 핸들러를 부트스트랩하는 짧은 어셈블리 코드입니다. 이 코드를 이해할 필요는 없습니다.

#### `exception.c`, `exception.h`

사용자 프로세스가 특권 연산이나 금지된 연산을 수행하면 커널로 트랩되어 `exception` 또는 `fault`로 들어옵니다. 이 파일들이 예외를 처리합니다. 현재는 모든 예외가 메시지를 출력하고 프로세스를 종료합니다. 프로젝트 2 해법 중 일부는 이 파일의 `page_fault()`를 수정해야 합니다.

#### `gdt.c`, `gdt.h`

x86-64는 세그먼트 아키텍처입니다. GDT(Global Descriptor Table)는 사용 중인 세그먼트를 기술하는 테이블이며, 이 파일들이 GDT를 설정합니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없을 것입니다. GDT가 어떻게 동작하는지 궁금하면 코드를 읽어보세요.

#### `tss.c`, `tss.h`

TSS(Task-State Segment)는 x86의 아키텍처적 태스크 전환에 쓰였습니다. 하지만 x86-64에서는 태스크 전환이 폐지되었습니다. 그럼에도 TSS는 **링 전환 시 스택 포인터를 찾기 위해** 여전히 존재합니다.

즉, 사용자 프로세스가 인터럽트 핸들러로 들어갈 때, 하드웨어는 커널 스택 포인터를 찾기 위해 TSS를 참조합니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없을 것입니다. TSS에 관심 있으면 코드를 읽어보세요.

### Using the File System

사용자 프로그램은 파일 시스템에서 적재되고, 구현해야 할 시스템 콜 상당수도 파일 시스템과 관련되므로 이번 프로젝트에서 파일 시스템 코드와 맞닿아야 합니다. 다만 이번 과제의 초점은 파일 시스템이 아니므로, `filesys` 디렉터리에 단순하지만 완전한 파일 시스템을 제공해 두었습니다. 사용법을 이해하려면 `filesys.h`와 `file.h` 인터페이스를 살펴보고, 특히 **제약이 많다**는 점을 기억하세요.

**이번 프로젝트에서는 파일 시스템 코드를 수정할 필요가 없으며, 수정하지 않는 것을 권장합니다.** 파일 시스템에 손대면 이번 과제의 초점에서 벗어나기 쉽습니다.

지금 파일 시스템 루틴을 올바르게 써 두면 **프로젝트 4**에서 파일 시스템 구현을 개선할 때 훨씬 수월해집니다. 그때까지는 아래 제약을 감수해야 합니다.

- 내부 동기화가 없습니다. 동시 접근은 서로 간섭합니다. **한 번에 하나의 프로세스만** 파일 시스템 코드를 실행하도록 동기화해야 합니다.
- 파일 크기는 생성 시점에 고정됩니다. 루트 디렉터리도 파일로 표현되므로, 생성 가능한 파일 수에도 한계가 있습니다.
- 파일 데이터는 **단일 extent**로 할당됩니다. 즉, 한 파일의 데이터는 디스크에서 연속된 섹터 범위를 차지해야 합니다. 따라서 사용이 반복되면 **외부 단편화**가 심각한 문제가 될 수 있습니다.
- 하위 디렉터리가 없습니다.
- 파일 이름은 14자로 제한됩니다.
- 연산 도중 시스템이 크래시하면 디스크가 자동으로 복구 불가능한 방식으로 손상될 수 있습니다. 어차피 파일 시스템 복구 도구도 없습니다.

**다음 한 가지는 중요한 특징입니다.**

`filesys_remove()`에 **Unix와 유사한 의미**가 구현되어 있습니다. 즉, 파일이 열린 상태에서 삭제되면 블록은 바로 해제되지 않으며, 해당 파일을 연 스레드가 모두 닫을 때까지 접근할 수 있습니다. 자세한 내용은 [Removing an Open File](https://casys-kaist.github.io/pintos-kaist/project2/FAQ.html)을 참고하세요.

프로젝트 1에서는 모든 테스트 프로그램이 이미 커널 이미지 안에 있었지만, 이번에는 **사용자 공간에서 돌아가는** 테스트 프로그램을 Pintos 가상 머신 안에 넣어야 합니다. 다행히 테스트 스크립트(`make check`)가 이 과정을 자동으로 처리하므로, 대부분의 경우 이 부분을 몰라도 됩니다. 다만 **개별 테스트를 직접 돌릴 때**는 아래 내용을 알면 큰 도움이 됩니다.

가상 머신에 파일을 넣으려면, 먼저 파일 시스템 파티션이 있는 시뮬레이션 디스크를 만들 수 있어야 합니다. `pintos-mkdisk` 프로그램이 그 역할을 합니다. `userprog/build` 디렉터리에서 `pintos-mkdisk filesys.dsk 2`를 실행하면, 2MB Pintos 파일 시스템 파티션이 들어 있는 `filesys.dsk`라는 시뮬레이션 디스크가 만들어집니다. 그다음 `pintos`에 `--fs-disk filesys.dsk`를 `pintos`와 `--` **사이에** 넘깁니다(예: `pintos --fs-disk filesys.dsk -- KERNEL_COMMANDS...`). `--`가 필요한 이유는 `--fs-disk`가 시뮬레이션 커널이 아니라 **pintos 스크립트** 옵션이기 때문입니다. 이후 커널 명령줄에 `-f -q`를 넘겨 파일 시스템 파티션을 포맷합니다: `pintos SCRIPT_COMMANDS -- -f -q`. `-f`는 포맷을 의미하고, `-q`는 포맷이 끝나자마자 Pintos를 종료합니다.

시뮬레이션 파일 시스템으로 파일을 넣고 빼려면 pintos의 `-p`(put)와 `-g`(get) 옵션을 씁니다. Pintos 파일 시스템으로 `file`을 복사하려면 `pintos -p file -- -q`를 사용합니다. Pintos 쪽 이름을 `newname`으로 바꾸려면 원본 파일명 뒤에 `:newname`을 붙입니다: `pintos -p file:newname -- -q`. VM에서 꺼내는 명령은 비슷하지만 `-p` 대신 `-g`를 씁니다.

이 옵션들은 커널 명령줄에 특수 명령 extract/append를 넘기고, 특수 시뮬레이션 "scratch" 파티션과 주고받으며 동작합니다. 궁금하면 `pintos` 스크립트와 `filesys/fsutil.c`를 보면 구현 세부를 알 수 있습니다.

아래는 파일 시스템 파티션이 있는 디스크를 만들고, 포맷한 뒤, 이번 프로젝트의 두 번째 테스트인 `args-single` 프로그램을 새 디스크에 복사하고, 인자 `onearg`를 넘겨 실행하는 과정을 한 번에 요약한 것입니다. (인자 전달은 구현하기 전까지는 동작하지 않습니다.) 테스트를 이미 빌드했고 현재 디렉터리가 `userprog/build`라고 가정합니다.

```
pintos-mkdisk filesys.dsk 10
pintos --fs-disk filesys.dsk -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

나중에 쓰거나 검사하기 위해 파일 시스템 디스크를 남겨 두고 싶지 않다면, 네 단계를 **한 줄**로 합칠 수도 있습니다. `--filesys-size=n` 옵션은 pintos 실행 동안만 존재하는, 대략 nMB 크기의 임시 파일 시스템 파티션을 만듭니다. Pintos 자동 테스트 스위트는 이 문법을 많이 사용합니다.

```
pintos --fs-disk=10 -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

### How User Programs Work

Pintos는 메모리에 들어가고 여러분이 구현한 시스템 콜만 사용하는 한, 일반 C 프로그램을 실행할 수 있습니다. 특히 **`malloc()`은 구현할 수 없습니다.** 이번 프로젝트에 필요한 시스템 콜만으로는 메모리 할당이 불가능하기 때문입니다. 또한 스레드 전환 시 FPU를 저장/복원하지 않으므로 **부동소수점 연산**을 쓰는 프로그램도 실행할 수 없습니다.

Pintos는 `userprog/process.c`에 제공된 로더로 ELF 실행 파일을 적재할 수 있습니다. ELF는 Linux, Solaris 등 많은 OS에서 객체 파일, 공유 라이브러리, 실행 파일에 쓰이는 형식입니다.

x86-64 ELF를 출력하는 어떤 컴파일러/링커로도 Pintos용 프로그램을 만들 수 있습니다(제공된 도구로 충분할 것입니다). 테스트 프로그램을 시뮬레이션 파일 시스템에 복사하기 전에는 Pintos가 유의미한 일을 거의 못 한다는 점을 곧 깨닫게 됩니다. 다양한 프로그램을 복사해 넣기 전에는 흥미로운 실험도 어렵습니다. 디버깅 중 `filesys.dsk`를 망가뜨리는 일이 가끔 있으니, 깨끗한 참조 디스크를 만들어 두었다가 필요할 때마다 덮어쓰는 방법도 고려해 보세요.

### Virtual Memory Layout

Pintos의 가상 메모리는 **사용자 가상 메모리**와 **커널 가상 메모리** 두 영역으로 나뉩니다. 사용자 가상 메모리는 가상 주소 `0`부터 `KERN_BASE`까지입니다(`KERN_BASE`는 `include/threads/vaddr.h`에 정의되어 있으며 기본값은 `0x8004000000`). 커널 가상 메모리는 나머지 가상 주소 공간을 차지합니다.

사용자 가상 메모리는 **프로세스마다** 다릅니다. 커널이 한 프로세스에서 다른 프로세스로 전환할 때는, 프로세서의 페이지 디렉터리 베이스 레지스터를 바꿔 사용자 가상 주소 공간도 함께 바꿉니다(`thread/mmu.c`의 `pml4_activate()` 참고). `struct thread`는 프로세스의 페이지 테이블을 가리키는 포인터를 담습니다.

커널 가상 메모리는 **전역**입니다. 어떤 사용자 프로세스나 커널 스레드가 실행 중이든 항상 같은 방식으로 매핑됩니다. Pintos에서 커널 가상 메모리는 `KERN_BASE`부터 물리 메모리에 **1:1**로 매핑됩니다. 즉, 가상 주소 `KERN_BASE`는 물리 주소 `0`에 대응하고, 가상 주소 `KERN_BASE + 0x1234`는 물리 주소 `0x1234`에 대응하며, 머신의 물리 메모리 크기까지 이어집니다.

사용자 프로그램은 자신의 사용자 가상 메모리만 접근할 수 있습니다. 커널 가상 메모리에 접근하려 하면 `userprog/exception.c`의 `page_fault()`에서 처리되는 페이지 폴트가 나고 프로세스는 종료됩니다. 커널 스레드는 커널 가상 메모리와, 사용자 프로세스가 실행 중이라면 그 실행 중인 프로세스의 사용자 가상 메모리에도 접근할 수 있습니다. 하지만 커널 안에서조차 **매핑되지 않은 사용자 가상 주소**에 접근하면 페이지 폴트가 납니다.

### Typical Memory Layout

개념적으로 각 프로세스는 사용자 가상 메모리를 원하는 대로 배치할 수 있지만, 실제로는 대략 아래와 같이 배치됩니다.

```
USER_STACK +----------------------------------+
           |             user stack           |
           |                 |                |
           |                 |                |
           |                 V                |
           |           grows downward         |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |           grows upward           |
           |                 ^                |
           |                 |                |
           |                 |                |
           +----------------------------------+
           | uninitialized data segment (BSS) |
           +----------------------------------+
           |     initialized data segment     |
           +----------------------------------+
           |            code segment          |
 0x400000  +----------------------------------+
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
       0   +----------------------------------+
```

이번 프로젝트에서는 사용자 스택 크기가 고정되어 있지만, 프로젝트 3에서는 스택이 자라도록 허용됩니다. 전통적으로는 시스템 콜로 BSS 크기를 조정할 수 있지만, 여러분은 그것을 구현할 필요가 없습니다.

Pintos에서 코드 세그먼트는 사용자 가상 주소 `0x400000`에서 시작합니다. 주소 공간 맨 아래에서 약 128MB 위에 해당하며, 이 값은 우분투에서 흔히 쓰이던 값일 뿐 특별한 의미는 없습니다.

링커는 링커 스크립트의 지시에 따라 사용자 프로그램의 메모리 배치를 정합니다. 링커 스크립트에 대해 더 알고 싶다면 링커 매뉴얼의 "Scripts" 장을 읽으세요(`info ld`).

특정 실행 파일의 배치를 보려면 `objdump`에 `-p` 옵션을 주면 됩니다.

### Accessing User Memory

시스템 콜 처리 과정에서 커널은 사용자 프로그램이 넘긴 포인터를 통해 메모리에 자주 접근해야 합니다. 이때 커널은 매우 조심해야 합니다. 사용자는 널 포인터, 매핑되지 않은 가상 메모리를 가리키는 포인터, 커널 가상 주소 공간(`KERN_BASE` 위)을 가리키는 포인터를 넘길 수 있습니다. 이런 잘못된 포인터는 **커널이나 다른 실행 중인 프로세스에 해를 끼치지 않고** 거절해야 하며, 위반한 프로세스를 종료하고 자원을 해제해야 합니다.

이를 올바르게 처리하는 합리적인 방법은 적어도 두 가지입니다.

**첫 번째 방법**은 사용자가 준 포인터의 유효성을 검사한 뒤 역참조하는 것입니다. 이 길을 택한다면 `thread/mmu.c`와 `include/threads/vaddr.h`의 함수들을 살펴보세요. 사용자 메모리 접근을 다루는 가장 단순한 방법입니다.

**두 번째 방법**은 사용자 포인터가 `KERN_BASE` 아래를 가리키는지만 확인한 뒤 역참조하는 것입니다. 잘못된 사용자 포인터는 "페이지 폴트"를 일으키고, `userprog/exception.c`의 `page_fault()` 코드를 수정해 처리할 수 있습니다. 프로세서 MMU를 활용하므로 보통 더 빠르고, 실제 커널(Linux 포함)에서도 자주 쓰입니다.

어느 쪽이든 **자원이 새지 않도록** 해야 합니다. 예를 들어 시스템 콜이 이미 락을 잡았거나 `malloc()`으로 메모리를 할당한 뒤 잘못된 사용자 포인터를 만나면, 그래도 락을 놓거나 메모리를 반드시 해제해야 합니다. 역참조 전에 포인터를 검증하면 이 경우가 비교적 단순합니다. 잘못된 포인터가 페이지 폴트를 일으키는 쪽은 더 까다롭습니다. 메모리 접근에서 에러 코드를 돌려줄 방법이 없기 때문입니다. 후자를 시도하려는 분을 위해 조금 도움이 되는 코드를 제공합니다.

```c
/* Reads a byte at user virtual address UADDR.
 * UADDR must be below KERN_BASE.
 * Returns the byte value if successful, -1 if a segfault
 * occurred. */
static int64_t
get_user (const uint8_t *uaddr) {
    int64_t result;
    __asm __volatile (
    "movabsq $done_get, %0\n"
    "movzbq %1, %0\n"
    "done_get:\n"
    : "=&a" (result) : "m" (*uaddr));
    return result;
}

/* Writes BYTE to user address UDST.
 * UDST must be below KERN_BASE.
 * Returns true if successful, false if a segfault occurred. */
static bool
put_user (uint8_t *udst, uint8_t byte) {
    int64_t error_code;
    __asm __volatile (
    "movabsq $done_put, %0\n"
    "movb %b2, %1\n"
    "done_put:\n"
    : "=&a" (error_code), "=m" (*udst) : "q" (byte));
    return error_code != -1;
}
```

이 함수들은 사용자 주소가 이미 `KERN_BASE` 아래임이 확인되었다고 가정합니다. 또한 **`page_fault()`를 수정해**, 커널에서 발생한 페이지 폴트가 `rax`를 -1로 만들고 이전 값을 `%rip`에 넣도록 해 두었다고 가정합니다.
