# Advanced Scheduler

시스템에서 실행 중인 작업의 평균 응답 시간을 줄이기 위해, 4.4BSD 스케줄러와 유사한 **MLFQS(multilevel feedback queue scheduler)** 를 구현하세요.

advanced scheduler도 priority scheduler처럼 우선순위를 기준으로 실행 스레드를 선택합니다.  
단, advanced scheduler는 **priority donation을 사용하지 않습니다**. 따라서 advanced scheduler를 시작하기 전에 (priority donation을 제외하고라도) priority scheduler를 먼저 동작시키는 것을 권장합니다.

Pintos 시작 시 스케줄링 정책을 선택할 수 있도록 구현해야 합니다.

- 기본: priority scheduler 활성
- 옵션: 커널 옵션 `-mlfqs`를 주면 4.4BSD scheduler 선택

`-mlfqs` 전달 시 `threads/thread.h`에 선언된 `thread_mlfqs`가 `true`로 설정됩니다. 이 설정은 `main()` 초반에 호출되는 `parse_options()`에서 처리됩니다.

4.4BSD scheduler가 활성화되면, 스레드는 더 이상 우선순위를 직접 제어하지 않습니다.

- `thread_create()`의 priority 인자는 무시
- `thread_set_priority()` 호출 무시
- `thread_get_priority()`는 scheduler가 계산한 현재 우선순위를 반환

advanced scheduler는 이후 프로젝트에서 사용되지 않습니다.

## 4.4BSD Scheduler

일반 목적 스케줄러의 목표는 서로 다른 요구를 가진 스레드들의 균형을 맞추는 것입니다.

- I/O 중심 스레드: 빠른 응답 시간 필요, CPU 시간은 적게 필요
- 계산 중심 스레드: CPU 시간 많이 필요, 빠른 응답 시간 요구는 낮음
- 혼합형 스레드: I/O와 계산 구간이 번갈아 나타남

잘 설계된 스케줄러는 이 요구를 동시에 만족시킬 수 있습니다.

Project 1에서는 본 부록의 스케줄러를 구현해야 합니다. 이는 [McKusick]에서 설명하는 MLFQS와 유사합니다.  
이 스케줄러는 우선순위별 ready queue를 여러 개 유지하며, 항상 가장 높은 우선순위의 비어 있지 않은 큐에서 스레드를 선택합니다. 해당 큐에 여러 스레드가 있으면 round robin으로 실행합니다.

스케줄러는 timer tick 수에 따라 여러 데이터를 주기적으로 갱신해야 합니다. 모든 갱신은 일반 커널 스레드가 실행되기 전에 수행되어야 합니다. 즉, 스레드가 증가된 `timer_ticks()` 값은 보지만 스케줄러 데이터는 옛값인 상태가 되면 안 됩니다.

4.4BSD scheduler에는 priority donation이 없습니다.

## Niceness

스레드 우선순위는 아래 공식으로 동적으로 계산됩니다. 또한 각 스레드는 `nice` 정수 값을 가집니다.

- `nice = 0`: 우선순위에 영향 없음
- 양수 (`최대 20`): 우선순위 하락, CPU 양보 경향
- 음수 (`최소 -20`): 다른 스레드의 CPU 시간을 더 가져오는 경향

초기 스레드 nice는 0입니다. 다른 스레드는 부모 스레드의 nice를 상속받습니다.

테스트에서 사용하는 아래 함수를 구현해야 합니다 (`threads/thread.c`에 skeleton 제공).

```c
int thread_get_nice (void);
```

- 현재 스레드의 nice 반환

```c
int thread_set_nice (int nice);
```

- 현재 스레드의 nice를 새 값으로 설정
- 새 값 기반으로 우선순위 재계산(아래 계산식 참고)
- 최고 우선순위가 아니면 즉시 yield

## Calculating Priority

스케줄러는 64개 우선순위(=64개 ready queue)를 가집니다.

- 범위: `0 (PRI_MIN)` ~ `63 (PRI_MAX)`
- 숫자가 낮을수록 낮은 우선순위

우선순위는 초기화 시 계산되고, 이후 모든 스레드에 대해 매 4 tick마다 재계산됩니다.

```
priority = PRI_MAX - (recent_cpu / 4) - (nice * 2)
```

여기서:

- `recent_cpu`: 최근 CPU 사용량 추정치
- `nice`: 스레드의 nice 값

결과는 소수점 버림(truncate)으로 정수화하고, 항상 `PRI_MIN ~ PRI_MAX` 범위로 보정해야 합니다.

이 공식은 최근 CPU를 많이 쓴 스레드의 우선순위를 낮춰 재할당을 늦추는 효과가 있습니다. 이는 starvation 방지의 핵심입니다.

## Calculating recent_cpu

`recent_cpu`는 스레드가 최근에 얼마나 CPU를 썼는지 측정합니다. 최근일수록 더 큰 가중치를 주기 위해 지수 가중 이동평균(EWMA)을 사용합니다.

일반 형태:

```
x(0) = f(0)
x(t) = a*x(t - 1) + (1 - a)*f(t)
a = k/(k + 1)
```

`recent_cpu` 초기값:

- 첫 스레드: `0`
- 그 외 새 스레드: 부모의 값 상속

업데이트 규칙:

- 매 timer interrupt마다 (idle thread 제외) 실행 중 스레드의 `recent_cpu += 1`
- 매 1초마다 모든 스레드(running/ready/blocked)에 대해:

```
recent_cpu = (2 * load_avg)/(2 * load_avg + 1) * recent_cpu + nice
```

테스트 조건상 `recent_cpu` 재계산은 반드시 아래 시점에서만 이뤄져야 합니다.

```
timer_ticks() % TIMER_FREQ == 0
```

주의:

- nice가 음수면 `recent_cpu`도 음수가 될 수 있음
- 음수 `recent_cpu`를 0으로 clamp 하면 안 됨
- 오버플로 방지를 위해 `(2*load_avg)/(2*load_avg+1)` 계수를 먼저 계산한 뒤 곱하는 방식을 권장

구현해야 할 함수:

```c
int thread_get_recent_cpu (void);
```

- 현재 스레드 `recent_cpu * 100`을 반올림해 정수 반환

## Calculating load_avg

`load_avg`(system load average)는 지난 1분간 runnable 스레드 수 평균을 추정합니다.  
`recent_cpu`와 마찬가지로 EWMA지만, 스레드별이 아닌 **시스템 전역 값**입니다.

- 부팅 시 초기값: `0`
- 이후 매 1초마다:

```
load_avg = (59/60) * load_avg + (1/60) * ready_threads
```

`ready_threads`는 업데이트 시점에 running 또는 ready 상태인 스레드 수(단, idle thread 제외)입니다.

역시 테스트 조건상 `load_avg` 갱신은 정확히 아래 시점에서만 해야 합니다.

```
timer_ticks() % TIMER_FREQ == 0
```

구현해야 할 함수:

```c
int thread_get_load_avg (void)
```

- 현재 시스템 `load_avg * 100`을 반올림해 정수 반환

## Summary

필수 계산식 요약(요구사항 전체를 대체하진 않음):

매 4 tick마다:

```
priority = PRI_MAX - (recent_cpu / 4) - (nice * 2)
```

매 tick마다(running thread):

```
recent_cpu += 1
```

매 1초마다(모든 스레드):

```
recent_cpu = (2 * load_avg) / (2 * load_avg + 1) * recent_cpu + nice
```

매 1초마다(시스템 전역):

```
load_avg = (59/60) * load_avg + (1/60) * ready_threads
```

## Fixed-Point Real Arithmetic

위 공식들에서 `priority`, `nice`, `ready_threads`는 정수지만 `recent_cpu`, `load_avg`는 실수입니다.  
Pintos 커널은 부동소수점 연산을 지원하지 않으므로 정수 기반 고정소수점(fixed-point)으로 실수 연산을 흉내 내야 합니다.

핵심 아이디어:

- 정수의 하위 비트를 소수부로 사용
- 예: signed 32-bit에서 하위 14비트를 소수부로 쓰면 17.14 형식
- 값 `x`는 실제로 `x / 2^14` 의미

`p.q` 형식에서 `f = 2^q`라 하면:

- 정수/실수 -> fixed-point: `value * f`
- fixed-point -> 정수(0 방향 반올림): `x / f`
- 최근접 반올림:
  - 양수: `(x + f/2) / f`
  - 음수: `(x - f/2) / f`

### 기본 연산

- `x + y`
- `x - y`
- `x + n*f`
- `x - n*f`
- `x * n`
- `x / n`

### fixed-point 곱셈

```c
((int64_t) x) * y / f
```

### fixed-point 나눗셈

```c
((int64_t) x) * f / y
```

64비트 연산을 쓰는 이유는 오버플로 방지입니다.

### 연산 표

| 연산 | C 구현 |
|---|---|
| n을 고정소수점으로 변환 | `n * f` |
| x를 정수로 변환(0 방향) | `x / f` |
| x를 정수로 변환(최근접) | `x >= 0 ? (x + f/2)/f : (x - f/2)/f` |
| x + y | `x + y` |
| x - y | `x - y` |
| x + n | `x + n * f` |
| x - n | `x - n * f` |
| x * y | `((int64_t) x) * y / f` |
| x * n | `x * n` |
| x / y | `((int64_t) x) * f / y` |
| x / n | `x / n` |
