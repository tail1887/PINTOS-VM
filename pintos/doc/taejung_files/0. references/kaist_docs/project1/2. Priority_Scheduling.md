# Priority Scheduling

Pintos에서 **priority scheduling**과 **priority donation**을 구현하세요.

현재 실행 중인 스레드보다 우선순위가 높은 스레드가 ready list에 추가되면, 현재 스레드는 즉시 CPU를 양보해야 합니다.  
마찬가지로 lock, semaphore, condition variable을 기다리는 스레드들 중에서는 **가장 높은 우선순위 스레드가 먼저 깨어나야** 합니다.

스레드는 언제든 자신의 우선순위를 올리거나 내릴 수 있습니다. 단, 우선순위를 낮춘 결과 더 이상 최고 우선순위가 아니게 되면 즉시 CPU를 양보해야 합니다.

스레드 우선순위 범위:

- `PRI_MIN (0)` ~ `PRI_MAX (63)`
- 숫자가 낮을수록 낮은 우선순위
- `0`이 최저, `63`이 최고

초기 우선순위는 `thread_create()` 인자로 전달됩니다. 특별한 이유가 없으면 `PRI_DEFAULT (31)`을 사용하세요.  
`PRI_` 매크로는 `threads/thread.h`에 정의되어 있으며 값은 변경하면 안 됩니다.

우선순위 스케줄링의 대표 문제는 **priority inversion(우선순위 역전)** 입니다.  
예를 들어 H(높음), M(중간), L(낮음) 스레드가 있을 때, H가 L이 가진 lock을 기다리고 M이 ready list에 있으면, L이 CPU를 못 받아 H도 영원히 실행되지 못할 수 있습니다.

이 문제의 부분 해결책이 **priority donation**입니다. L이 lock을 들고 있는 동안 H가 자신의 우선순위를 L에게 기부하고, L이 lock을 해제(그리고 H가 획득)하면 기부를 회수합니다.

priority donation 구현 시 반드시 다음을 처리해야 합니다.

- 여러 스레드가 하나의 스레드에 우선순위를 기부하는 **multiple donation**
- 연쇄 대기 상황의 **nested donation**  
  (예: H가 M의 lock을 기다리고, M은 L의 lock을 기다리는 경우 M과 L 모두 H 우선순위로 상승)

필요하면 nested donation 깊이에 합리적인 제한(예: 8단계)을 둘 수 있습니다.

필수 조건:

- **lock에 대해서는 priority donation을 반드시 구현**
- 다른 동기화 구조체에 대한 donation은 필수 아님
- priority scheduling은 모든 경우에 구현해야 함

추가로 아래 함수들을 구현하세요 (`threads/thread.c`에 skeleton 제공).

```c
void thread_set_priority (int new_priority);
```

- 현재 스레드 우선순위를 `new_priority`로 설정
- 설정 후 최고 우선순위가 아니면 즉시 yield

```c
int thread_get_priority (void);
```

- 현재 스레드 우선순위 반환
- donation이 있으면 더 높은(기부된) 우선순위를 반환

스레드가 다른 스레드의 우선순위를 직접 수정하는 별도 인터페이스는 구현할 필요 없습니다.

참고로 priority scheduler는 이후 프로젝트에서 사용되지 않습니다.
