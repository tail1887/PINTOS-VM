# Project1: Threads

이번 과제에서는 최소 기능만 갖춘 스레드 시스템이 제공됩니다. 여러분의 목표는 이 시스템의 기능을 확장하면서 동기화 문제를 더 깊이 이해하는 것입니다.  
작업은 주로 `threads` 디렉터리에서 진행하고, 일부는 `devices` 디렉터리에서 수행합니다. 컴파일은 `threads` 디렉터리에서 진행해야 합니다.

프로젝트 설명을 본격적으로 읽기 전에 `Synchronization` 자료를 최소한 한 번 훑어보세요.

## Background

### Understanding Threads

첫 단계는 초기 스레드 시스템 코드를 읽고 이해하는 것입니다. Pintos는 이미 다음을 구현해 둔 상태입니다.

- 스레드 생성 / 종료
- 스레드 간 전환을 위한 단순 스케줄러
- 동기화 기본 요소(semaphore, lock, condition variable, optimization barrier)

코드 일부는 처음에 다소 낯설 수 있습니다. 아직 기본 시스템을 빌드/실행해보지 않았다면 먼저 실행해 보세요. 소스 코드를 읽고, 필요하면 `printf()`를 추가한 뒤 재컴파일해서 실행 순서를 확인해도 좋습니다. 디버거로 커널을 실행해 브레이크포인트를 걸고, 한 줄씩 따라가며 데이터 상태를 확인하는 방법도 매우 유용합니다.

스레드를 생성한다는 것은 스케줄 가능한 새로운 실행 컨텍스트를 만드는 것입니다. `thread_create()`에 전달한 함수가 해당 컨텍스트에서 실행됩니다. 스레드가 처음 스케줄되어 실행될 때, 그 함수의 시작부터 실행되고 함수가 반환되면 스레드는 종료됩니다. 즉, 각 스레드는 Pintos 내부에서 동작하는 작은 프로그램처럼 볼 수 있고, `thread_create()`에 넘긴 함수는 그 스레드의 `main()` 역할을 합니다.

어떤 시점이든 실행 중인 스레드는 정확히 하나이며, 나머지는 비활성 상태입니다. 다음에 어떤 스레드를 실행할지는 스케줄러가 결정합니다. (실행 가능한 스레드가 없다면 `idle()`에 구현된 특수 idle 스레드가 실행됩니다.) 동기화 프리미티브는 한 스레드가 다른 스레드를 기다려야 할 때 문맥 전환을 유도할 수 있습니다.

문맥 전환의 핵심 메커니즘은 `threads/thread.c`의 `thread_launch()`에 있습니다. (완벽히 이해하지 않아도 됩니다.) 현재 스레드의 상태를 저장하고, 전환 대상 스레드의 상태를 복원합니다.

GDB로 문맥 전환을 천천히 추적해 보세요. `schedule()`에 브레이크포인트를 걸고 single-step으로 따라가면 도움이 됩니다. 각 스레드의 주소/상태, 콜 스택의 함수들을 함께 추적해 보세요. 한 스레드가 `do_iret()`의 `iret`를 실행하면 다른 스레드가 실행을 시작하는 지점을 관찰할 수 있습니다.

> ⚠️ 주의  
> Pintos에서 각 스레드는 4kB에 약간 못 미치는 작은 고정 크기 스택을 가집니다. 커널이 스택 오버플로를 감지하려고 시도하지만 완벽하지 않습니다.  
> `int buf[1000];` 같은 큰 비정적 지역 변수를 선언하면 원인 파악이 어려운 커널 패닉이 발생할 수 있습니다.  
> 큰 메모리는 스택 대신 페이지 할당기/블록 할당기를 사용하세요.

## Source Files

`threads` 및 `include/threads` 디렉터리 주요 파일 요약입니다. 대부분 수정할 필요는 없지만, 어디를 봐야 하는지 감을 잡는 데 도움이 됩니다.

### threads 코드

- `loader.S`, `loader.h`  
커널 로더. BIOS가 메모리에 올리는 512바이트 코드/데이터로 구성됩니다. 디스크에서 커널을 찾아 메모리에 적재하고 `start.S`의 `bootstrap()`으로 점프합니다. 보통 수정할 필요 없습니다. `start.S`는 메모리 보호와 64비트 long mode 진입을 위한 초기 설정을 수행하며, 로더와 달리 커널 일부입니다.
- `kernel.lds.S`  
커널 링크 스크립트. 커널 로드 주소를 설정하고 `start.S`가 커널 이미지 앞쪽에 위치하도록 배치합니다. 보통 수정할 필요는 없습니다.
- `init.c`, `init.h`  
`main()`을 포함한 커널 초기화 코드. 무엇이 초기화되는지 확인하려면 `main()`은 꼭 읽어보세요. 필요하면 초기화 코드를 여기에 추가할 수 있습니다.
- `thread.c`, `thread.h`  
기본 스레드 지원 코드. 과제의 핵심 작업이 많이 이 파일들에서 이뤄집니다. `thread.h`의 `struct thread`는 4개 프로젝트 전반에서 수정할 가능성이 큽니다.
- `palloc.c`, `palloc.h`  
페이지 할당기(4kB 단위 메모리 할당).
- `malloc.c`, `malloc.h`  
커널용 `malloc()/free()` 단순 구현.
- `interrupt.c`, `interrupt.h`  
기본 인터럽트 처리 및 인터럽트 on/off 제어 함수.
- `intr-stubs.S`, `intr-stubs.h`  
저수준 인터럽트 처리 어셈블리 코드.
- `synch.c`, `synch.h`  
semaphore/lock/condition variable/optimization barrier 구현. 모든 프로젝트에서 동기화에 사용합니다.
- `mmu.c`, `mmu.h`  
x86-64 페이지 테이블 연산 함수. lab1 이후 더 자주 보게 됩니다.
- `io.h`  
I/O 포트 접근 함수. 주로 `devices` 코드에서 사용됩니다.
- `vaddr.h`, `pte.h`  
가상 주소/페이지 테이블 엔트리 관련 함수/매크로. Project 3에서 중요도가 올라갑니다.
- `flags.h`  
x86-64 flags 레지스터 일부 비트 정의 매크로.

## Synchronization

올바른 동기화는 이 프로젝트 해법의 핵심입니다.

인터럽트를 끄면 동시성이 사라져 race condition이 없어지므로, 모든 동기화 문제를 이렇게 해결하고 싶어질 수 있습니다. 하지만 그렇게 하면 안 됩니다. 대부분의 동기화는 semaphore/lock/condition variable로 해결하세요.

Pintos에서 인터럽트 비활성화가 가장 적합한 경우는 **커널 스레드와 인터럽트 핸들러가 공유하는 데이터 조율**입니다. 인터럽트 핸들러는 sleep할 수 없어 lock을 획득할 수 없기 때문입니다. 따라서 이 경우 커널 스레드 측에서 인터럽트를 끄고 보호해야 합니다.

이번 프로젝트에서는 인터럽트 핸들러가 일부 스레드 상태에 접근합니다.

- Alarm Clock: 타이머 인터럽트가 sleep 중인 스레드를 깨움
- Advanced Scheduler: 타이머 인터럽트가 전역/스레드별 변수 일부 접근

커널 스레드에서 이 변수들에 접근할 때는 타이머 인터럽트 간섭을 막기 위해 인터럽트를 비활성화해야 합니다.

인터럽트를 끄는 구간은 반드시 최소화하세요. 너무 오래 끄면 timer tick/input event 손실이나 인터럽트 지연 증가로 시스템 반응성이 떨어질 수 있습니다.

`synch.c`의 동기화 프리미티브 자체도 인터럽트 비활성화 기반으로 구현됩니다. 필요 시 비활성화 구간을 늘릴 수는 있지만, 여전히 최소화하는 것이 원칙입니다.

디버깅 목적으로 인터럽트 비활성화를 사용할 수는 있지만, 제출 전에는 제거해야 합니다. (주석 처리만 해두면 코드 가독성이 크게 떨어질 수 있습니다.)

> 제출 코드에는 busy waiting이 없어야 합니다.  
> `thread_yield()`를 반복 호출하는 타이트 루프도 busy waiting입니다.

## Development Suggestions

과거에는 많은 팀이 과제를 쪼개 각자 마감 직전에 합치는 방식을 사용했지만, 이는 권장되지 않습니다. 변경 충돌로 인해 마감 직전 디버깅 폭탄이 자주 발생하고, 심한 경우 컴파일/부팅조차 안 되는 상태로 제출되는 일이 있었습니다.

대신 git 같은 버전 관리 시스템을 활용해 **초기에, 자주 통합**하는 방식을 권장합니다. 이렇게 하면 변경 사항을 서로 빠르게 공유하고, 버그가 들어온 시점을 추적해 안정 버전으로 쉽게 되돌아갈 수 있습니다.

이번 프로젝트와 이후 프로젝트에서도 이해하기 어려운 버그를 반드시 만나게 됩니다. 그럴 때는 디버깅 도구 부록을 다시 읽어보세요. 특히 backtrace 섹션은 커널 패닉/assertion failure에서 원인을 추적하는 데 매우 큰 도움이 됩니다.
