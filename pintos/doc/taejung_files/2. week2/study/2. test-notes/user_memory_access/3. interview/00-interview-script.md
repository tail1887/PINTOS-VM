# User Memory Access 면접 대본

## Q1: "User Memory Access가 뭐 하는 거예요?"

### 답변 (30초 - 기본 설명)

User Memory Access는 시스템 콜에서 사용자 프로세스가 전달한 포인터가 유효한지 검증하고, 안전하게 메모리를 복사하는 기능입니다.

예를 들어 `write()` 시스템 콜에서 사용자가 전달한 버퍼 포인터가 커널 메모리를 가리키거나 잘못된 주소라면, 이를 검증해서 프로세스를 안전하게 종료시키는 것입니다.

이게 없으면 악의적인 사용자 코드가 커널 메모리를 읽거나 쓰면서 시스템 전체를 망가뜨릴 수 있습니다.

---

## Q2: "그런데 구체적으로 어떤 단계를 거쳐요?"

### 답변 (2-3분 - 단계별 설명)

**3가지 단계로 나뉩니다.**

### 단계 1: 포인터 검증 단계

시스템 콜 핸들러에서 사용자가 전달한 포인터들을 `is_user_vaddr()`로 사용자 주소 공간인지 확인합니다. 추가로 `validate_user_memory()`로 메모리 영역 전체가 접근 가능한지 검사합니다.

이 과정에서 중요한 건:
- NULL 포인터는 즉시 거부
- 커널 주소 공간 침범은 금지
- 메모리 영역이 페이지 경계를 넘지 않는지 확인

### 단계 2: 안전 복사 단계

검증된 포인터를 사용해서 `copy_in()`이나 `copy_out()`으로 메모리를 복사합니다. 이 함수들은 페이지 폴트가 발생할 수 있으므로 예외 처리로 감싸서 안전하게 복사합니다.

문자열 복사의 경우 `strlcpy()` 같은 함수로 버퍼 오버플로를 방지합니다.

### 단계 3: 오류 처리 단계

복사 중 페이지 폴트가 발생하면 프로세스를 종료시키고, 성공하면 정상적으로 시스템 콜을 완료합니다.

---

## Q3: "포인터 검증 부분을 좀 더 자세히 설명해줄래요?"

### 답변 (1-2분 - 검증 로직 설명)

네, 이 부분이 가장 중요합니다. 구체적인 예로 설명하겠습니다.

`write(fd, buffer, size)` 시스템 콜에서 `buffer` 포인터를 검증한다고 하면:

```c
// 단계 1: 기본 검증
if (buffer == NULL || !is_user_vaddr(buffer))
    goto kill_process;

// 단계 2: 영역 검증
if (!validate_user_memory(buffer, size, true))  // true = 쓰기 권한 필요
    goto kill_process;

// 단계 3: 안전 복사
if (!copy_in(kernel_buffer, buffer, size))
    goto kill_process;  // 페이지 폴트 발생 시 종료
```

`is_user_vaddr()`는 주소가 `0x8004000000` (PHYS_BASE)보다 작은지 확인하고, `validate_user_memory()`는:
- 주소 범위가 사용자 공간 내인지
- 페이지가 실제로 매핑되어 있는지
- 접근 권한이 맞는지 확인합니다.

이렇게 하면 잘못된 포인터로 인한 시스템 크래시를 방지할 수 있습니다.

---

## Q4: "안전 복사가 왜 필요하죠?"

### 답변 (30초 - 메모리 보호)

사용자 메모리에 직접 접근하면 페이지 폴트가 발생할 수 있는데, 커널 모드에서 페이지 폴트가 나면 시스템 전체가 다운됩니다.

`copy_in()`/`copy_out()`은 예외 핸들러로 감싸서 페이지 폴트를 catch하고, 프로세스만 종료시키는 안전장치입니다.

---

## Q5: "구현 중에 가장 어려웠던 부분이 뭐예요?"

### 답변 (1-2분 - 어려운 점과 해결)

**두 가지가 있었습니다.**

### 어려운 점 1: 페이지 폴트 처리

`copy_in()`에서 memcpy를 그냥 호출하면 페이지 폴트 시 커널 패닉이 나는데, 이를 예외 핸들러로 감싸서 프로세스만 죽이도록 하는 게 까다로웠습니다.

해결책은 `process_kill_current()` 같은 함수를 예외 핸들러에 연결해서, 페이지 폴트 시점에 프로세스를 정리하고 종료시키는 거였습니다.

### 어려운 점 2: 경계 케이스

NULL 포인터, 페이지 경계에 걸친 주소, 매우 큰 size 값 등 다양한 엣지 케이스를 모두 처리해야 했습니다.

특히 `strlcpy()` 구현에서 버퍼 오버플로를 방지하면서도 NULL terminate를 제대로 하는 게 중요했습니다.

## Q6: "어떻게 테스트했어요?"

### 답변 (1-2분 - 테스트 전략)

**다양한 악의적인 입력으로 테스트했습니다.**

- `bad-read`, `bad-write`: 잘못된 주소로 읽기/쓰기 시도
- `bad-jump`: 잘못된 함수 포인터로 점프
- `syscall-bad-ptr`: 시스템 콜에 잘못된 포인터 전달
- `boundary-copy`: 페이지 경계에 걸친 복사

각 테스트에서 프로세스가 제대로 종료되는지, 그리고 시스템이 안정적인지 확인했습니다.

---

## Q7: "버그가 있었던 적이 있나요?"

### 답변 (1-2분 - 버그 경험)

네, `validate_user_memory()`에서 페이지 권한 체크를 빼먹은 적이 있었습니다.

읽기 전용 페이지에 쓰기를 시도해도 검증을 통과해서, 실제 memcpy에서 페이지 폴트가 나서 프로세스가 죽었습니다.

이걸 고치기 위해 페이지 테이블을 직접 조회해서 권한을 확인하는 코드를 추가했습니다.

---

## Q8: "실제 OS에서는 어떻게 다른가요?"

### 답변 (1-2분 - 실용성)

Linux에서는 비슷한 개념으로 `copy_from_user()`/`copy_to_user()` 함수가 있고, Windows에서는 SEH(Structured Exception Handling)로 비슷한 보호를 합니다.

하지만 Pintos는 교육용이라 더 엄격하게 검증하는데, 실제 OS는 성능 최적화를 위해 일부 검증을 생략하기도 합니다.

---

## Q9: "심화: 메모리 매핑이 어떻게 작동하나요?"

### 답변 (1-2분 - 심화 개념)

가상 메모리 시스템에서 사용자 주소는 페이지 테이블을 통해 물리 주소로 변환됩니다.

`is_user_vaddr()`는 주소 범위만 체크하지만, `validate_user_memory()`는 실제로 페이지 테이블을 조회해서:
- PTE(PTE)가 존재하는지
- 접근 권한이 맞는지
- 페이지가 물리 메모리에 있는지 확인합니다.

이게 없으면 dangling pointer 같은 문제가 생길 수 있습니다.

## Q10: "이 코드 리뷰해볼래요?"

### 답변 (2-3분 - 코드 리뷰)

```c
bool validate_user_memory(const void *uaddr, size_t size, bool write) {
    // 주소 범위 검증
    if (!is_user_vaddr(uaddr) || !is_user_vaddr(uaddr + size))
        return false;

    // 페이지별 검증
    for (uintptr_t addr = pg_round_down(uaddr);
         addr < pg_round_up(uaddr + size);
         addr += PGSIZE) {
        // 페이지 테이블 조회
        if (pml4_get_page(thread_current()->pml4, (void *)addr) == NULL)
            return false;
    }

    return true;
}
```

**좋은 점**: 페이지 단위로 철저히 검증
**개선점**: 권한 체크가 빠져있음 (읽기/쓰기 권한 확인 필요)

## 🎯 30초 버전 (긴급 요약)

User Memory Access는 시스템 콜에서 사용자 포인터를 검증하고 안전 복사하는 기능입니다.

포인터가 유효한지 확인하고, 페이지 폴트 시 프로세스를 죽여서 커널 보호합니다.

이 기능이 없으면 사용자 코드가 커널을 망가뜨릴 수 있습니다.

## 📝 추가 참고: 자주 들어오는 심화 질문

### "포인터 검증이 왜 이렇게 엄격한가요?"
커널 보호 때문입니다. 사용자 코드가 잘못된 포인터로 시스템 콜을 호출하면 커널 메모리를 오염시킬 수 있어요.

### "copy_in/copy_out이 memcpy와 뭐가 다른가요?"
예외 처리입니다. memcpy는 페이지 폴트 시 커널 패닉이 나지만, copy_in은 프로세스만 죽입니다.

### "모든 시스템 콜에 이 검증이 필요한가요?"
네, 사용자 입력을 받는 모든 시스템 콜에 필요합니다. 특히 포인터를 받는 read/write/open 등이요.

결론적으로, User Memory Access는 커널의 안전성을 위한 필수 기능입니다.

### "왜 사용자 메모리에 직접 접근하면 페이지 폴트가 발생해?"
좋은 질문입니다! 가상 메모리 시스템의 핵심 개념입니다.

**페이지 폴트 발생 이유**:
사용자 메모리는 **가상 메모리**이므로, 실제 물리 메모리에 매핑되어 있지 않거나 권한이 없을 수 있습니다.

**구체적 시나리오**:
1. **Unmapped 페이지**: 사용자가 `malloc()`하지 않은 주소 (예: `0x12345678`)를 전달하면, 그 가상 주소에 물리 페이지가 할당되지 않아 페이지 폴트 발생
2. **권한 부족**: 읽기 전용 페이지에 쓰기를 시도하거나, 실행 불가 페이지에 접근
3. **NULL 포인터**: `0x0` 주소는 의도적으로 매핑하지 않아 페이지 폴트 유발
4. **페이지 경계 초과**: 할당된 메모리 범위를 넘어서 접근

**커널 모드에서 왜 문제인가?**
- 시스템 콜 중 커널은 사용자 프로세스의 페이지 테이블을 사용하므로 사용자 메모리에 접근 가능
- 하지만 잘못된 주소에 접근하면 **페이지 폴트 인터럽트** 발생
- 커널 모드에서 페이지 폴트는 시스템 전체 크래시 유발 (블루스크린/커널 패닉)

**해결책**:
`copy_in()`/`copy_out()`은 예외 핸들러로 감싸서 페이지 폴트를 catch하고, 프로세스만 종료시킵니다.

```c
// 의사코드
bool copy_in(void *dst, const void *src, size_t size) {
    try {
        memcpy(dst, src, size);
        return true;
    } catch (page_fault) {
        process_kill_current();  // 프로세스만 죽임
        return false;
    }
}
```

결론적으로, 사용자 메모리는 언제든지 유효하지 않을 수 있으므로, 직접 접근 대신 안전한 복사 함수를 사용해야 합니다.
