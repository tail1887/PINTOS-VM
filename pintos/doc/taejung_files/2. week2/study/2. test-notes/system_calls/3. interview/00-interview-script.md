# System Calls 면접 대본

## Q1: "System Call이 뭐예요?"

### 답변 (30초 - 기본 설명)

System Call은 사용자 프로그램이 커널 기능을 요청할 때 사용하는 인터페이스입니다.

사용자 프로세스는 직접 하드웨어나 파일 시스템에 접근할 수 없기 때문에, `read`, `write`, `open`, `exit` 같은 요청을 커널에게 전달하고 커널이 대신 처리합니다.

---

## Q2: "그럼 System Call은 어떻게 동작해요?"

### 답변 (2-3분 - 흐름 설명)

1. 사용자 모드에서 시스템 콜을 호출합니다.
2. 트랩/인터럽트를 통해 CPU가 커널 모드로 전환됩니다.
3. 커널은 시스템 콜 번호를 읽고, 적절한 핸들러로 dispatch 합니다.
4. 인자 검증 후 작업을 수행합니다.
5. 결과를 사용자 공간으로 반환하고 다시 사용자 모드로 복귀합니다.

이 흐름에서 중요한 건 `사용자 주소 검증`, `정확한 dispatch`, `권한 확인`입니다.

---

## Q3: "인자는 어떻게 전달하고 검증하나요?"

### 답변 (1-2분 - 인자 전달/검증)

x86-64에서는 시스템 콜 인자가 레지스터에 들어옵니다. 예를 들어 `read(fd, buffer, size)`는 `RDI`, `RSI`, `RDX`에 각각 들어갑니다.

커널에서는 이 인자들을 그대로 사용하지 않고, 특히 사용자 포인터는 반드시 검증합니다.

```c
if (!is_user_vaddr(buffer) || !validate_user_memory(buffer, size, false))
    return -1;
```

이유는 사용자 주소가 잘못되어 있으면 페이지 폴트가 날 수 있기 때문입니다.

---

## Q4: "트랩과 인터럽트는 어떻게 연결되나요?"

### 답변 (30초 - 하드웨어 연동)

시스템 콜은 CPU의 트랩(software interrupt) 명령으로 발생합니다.

Pintos에서는 `syscall` 인터럽트 벡터를 정의하고, 해당 벡터에서 커널 진입과 시스템 콜 dispatch를 처리합니다.

이때 `intr_frame`은 사용자 모드 레지스터 상태를 저장하므로, 복귀 시점에 정확히 복원해야 합니다.

---

## Q5: "구현하면서 가장 어려웠던 부분은 뭐였어요?"

### 답변 (1-2분 - 어려움과 해결)

가장 어려운 부분은 **악의적인 사용자 입력 처리**였습니다.

특히 `buffer` 포인터가 잘못된 경우와 `size`가 너무 큰 경우를 모두 안전하게 처리해야 했습니다. 이를 위해 `validate_user_memory()`를 철저히 구현하고, `copy_in`/`copy_out` 같은 안전 복사 함수를 사용했습니다.

---

## Q6: "어떻게 테스트했어요?"

### 답변 (1-2분 - 테스트 전략)

- `read`/`write` 정상 경로
- 잘못된 파일 디스크립터 테스트
- NULL/잘못된 포인터 테스트
- 권한 없는 접근 테스트
- 경계값(페이지 경계) 테스트

이 테스트들을 통해 시스템 콜이 커널을 죽이지 않고 프로세스만 안전하게 종료되는지 확인했습니다.

---

## Q7: "버그가 있었던 적이 있나요?"

### 답변 (1-2분 - 버그 경험)

네, 초기에는 시스템 콜 번호를 잘못 읽어서 잘못된 핸들러로 dispatch 되는 문제가 있었습니다.

이 버그는 인터럽트 핸들러에서 `syscall_number`를 정확히 읽고, `switch-case`를 통해 안정적으로 분기하도록 수정하면서 해결했습니다.

---

## Q8: "실제 OS와 비교하면 어떤 차이가 있나요?"

### 답변 (1-2분 - 실용성)

Linux에서도 기본 개념은 동일합니다. `syscall` 트랩을 이용해 커널 모드로 전환하고, 사용자 인자를 검증합니다.

다만 Linux는 성능 최적화를 위해 추가적인 캐시와 fast path를 사용하고, 권한/모드 전환을 더 복잡하게 관리합니다.

---

## Q9: "심화: 시스템 콜 테이블은 어떻게 구성돼요?"

### 답변 (1-2분 - 심화 개념)

시스템 콜 테이블은 시스템 콜 번호를 함수 포인터로 매핑하는 배열입니다.

각 시스템 콜 번호는 고정된 인덱스로 사용되고, 커널은 해당 인덱스를 통해 올바른 구현 함수로 jump 합니다.

이 방식은 새로운 시스템 콜 추가 시 테이블에 항목만 추가하면 되기 때문에 확장성이 좋습니다.

---

## Q10: "코드 리뷰해볼래요?"

### 답변 (2-3분 - 코드 리뷰)

```c
switch (syscall_number) {
    case SYS_READ:
        f->R.rax = sys_read(f->R.rdi, (void *) f->R.rsi, f->R.rdx);
        break;
    case SYS_WRITE:
        f->R.rax = sys_write(f->R.rdi, (const void *) f->R.rsi, f->R.rdx);
        break;
    default:
        f->R.rax = -1;
        break;
}
```

**좋은 점**: 명료한 dispatch 구조
**개선점**: 인자 검증과 에러 처리를 더 분리하면 유지보수가 쉬워짐

## 🎯 30초 버전 (긴급 요약)

System Call은 사용자 프로그램이 커널에 작업을 요청하는 인터페이스입니다.

커널은 트랩을 통해 진입하고, 인자 검증 후 적절한 핸들러로 dispatch 하며 결과를 반환합니다.

이 기능이 없으면 사용자 프로그램이 커널을 직접 건드려 시스템 안정성이 깨집니다.

## 📝 추가 참고: 자주 들어오는 심화 질문

### "시스템 콜 번호는 어디서 오는 거예요?"
사용자 라이브러리나 어셈블리 코드가 `syscall` 명령을 실행할 때 레지스터에 번호를 넣습니다.

### "왜 사용자 포인터를 꼭 검증해야 하나요?"
잘못된 포인터는 페이지 폴트를 유발해서 커널을 크래시시킬 수 있습니다.

### "모든 시스템 콜에 이 검증이 필요해요?"
네, 사용자 인자를 받는 모든 시스템 콜에서 필요합니다. 특히 메모리 포인터를 받는 경우는 반드시 검증해야 합니다.

결론적으로, System Calls는 커널 안정성과 사용자-커널 인터페이스의 핵심입니다.
