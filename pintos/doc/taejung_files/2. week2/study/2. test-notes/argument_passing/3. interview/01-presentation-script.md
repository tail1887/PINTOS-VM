# Argument Passing 발표 대본

## 발표 목표

이 발표는 Argument Passing을 개념만 설명하는 것이 아니라, 실제 구현 코드인 `pintos/userprog/process.c`를 따라가면서 왜 이 순서로 처리했는지 설명하는 흐름입니다.

핵심 메시지는 다음 한 문장입니다.

> 커맨드라인 문자열을 안전하게 파싱하고, 사용자 스택에 `argc/argv` 구조를 만든 뒤, x86-64 호출 규약에 맞게 `RDI`, `RSI`, `RSP`를 세팅해서 유저 프로그램이 `_start(argc, argv)`로 정상 시작하게 만드는 과정입니다.

---

## 1. 도입부

안녕하세요. 저는 이번에 구현한 Argument Passing 부분을 설명드리겠습니다.

Argument Passing은 사용자가 입력한 커맨드라인을 사용자 프로그램의 `main(int argc, char **argv)`가 읽을 수 있는 형태로 전달하는 기능입니다.

예를 들어 `echo hello world`라는 명령이 들어오면, 단순히 문자열 하나를 넘기는 것이 아니라 `argc`는 3, `argv[0]`은 `"echo"`, `argv[1]`은 `"hello"`, `argv[2]`는 `"world"`가 되도록 사용자 스택과 레지스터를 맞춰줘야 합니다.

이 기능은 크게 세 단계로 나눠서 구현했습니다.

1. 커맨드라인 파싱
2. 사용자 스택에 문자열과 포인터 배열 배치
3. 레지스터 세팅 후 유저 모드 진입

이제 코드 흐름을 보면서 설명드리겠습니다.

---

## 2. 전체 흐름: `process_exec()`

먼저 `process_exec()` 함수부터 보겠습니다.

이 함수는 현재 프로세스를 새로운 사용자 프로그램 이미지로 교체하는 역할을 합니다. Argument Passing도 결국 새 프로그램을 실행하기 직전에 준비해야 하므로, 전체 흐름은 `process_exec()` 안에서 시작됩니다.

처음에는 `file_name`이 NULL인지, 길이가 한 페이지를 넘어가는지, 공백만 있는 입력인지 검사합니다.

여기서 중요한 점은 실행 파일 이름이 아예 없으면 이후에 `load()`를 호출할 수 없다는 것입니다. 그래서 파싱 전에 기본 입력 검증을 먼저 합니다.

그 다음 `cmd_line`이라는 새 페이지를 할당하고, 원본 커맨드라인을 복사합니다.

이렇게 복사하는 이유는 `strtok_r()`이 문자열 내부에 `'\0'`을 넣으면서 버퍼를 직접 수정하기 때문입니다. 원본을 그대로 건드리면 이후 경로에서 문제가 생길 수 있으므로, 쓰기 가능한 커널 버퍼를 따로 만든 뒤 파싱합니다.

또 하나 신경 쓴 부분은 `argv` 배열입니다. 처음에는 `char *argv[ARG_MAX]`처럼 지역 배열로 둘 수도 있지만, PintOS의 thread 구조는 4KB 페이지 안에 thread 구조체와 커널 스택이 같이 들어갑니다. 그래서 큰 지역 배열을 여러 개 쌓으면 커널 스택을 압박할 수 있습니다.

따라서 현재 구현에서는 `argv`도 `palloc_get_page()`로 페이지를 할당해서 관리합니다. 실패 경로에서는 `file_name`, `cmd_line`, `argv`를 각각 해제하도록 했습니다.

---

## 3. 1단계: 커맨드라인 파싱

다음은 `parse_command_line_args()`입니다.

이 함수의 역할은 `cmd_line` 문자열을 공백 기준으로 토큰화해서 `argc`와 `argv`를 채우는 것입니다.

첫 번째로 `strtok_r(cmd_line, " ", &save_ptr)`를 호출해서 첫 토큰을 가져옵니다. 첫 토큰이 없으면 실행 파일명이 없다는 뜻이기 때문에 바로 실패 처리합니다.

첫 토큰이 있으면 `argv[argc]`에 저장하고 `argc`를 증가시킵니다. 이후 반복문에서 `strtok_r(NULL, " ", &save_ptr)`를 호출하면서 나머지 인자들을 순서대로 저장합니다.

여기서 `ARG_MAX` 검사도 같이 들어갑니다. 토큰 개수가 배열 한계를 넘으면 더 이상 안전하게 저장할 수 없기 때문에 실패 처리합니다.

이 단계의 핵심은 스택을 아직 만들지 않는다는 것입니다. 파싱 단계는 오직 `argc`와 `argv[]`라는 중간 결과를 만드는 역할만 합니다.

이후 `finalize_parsed_args()`에서 `argv` 자체가 NULL인지, `argc`가 0 이하인지, `argv[0]`이 없는지 확인합니다. 여기까지 통과하면 `argv[0]`은 실행 파일명으로 사용할 수 있고, 전체 `argv`는 스택 빌더에 넘길 수 있는 상태가 됩니다.

---

## 4. 2단계: `load()` 이후 스택 구성

파싱이 끝나면 `process_exec()`는 기존 주소 공간을 정리하고 `load(argv[0], &_if)`를 호출합니다.

여기서 `argv[0]`만 넘기는 이유는 `load()`가 ELF 파일을 로드할 때 필요한 값은 실행 파일명뿐이기 때문입니다. 예를 들어 `echo hello world`라면 실제로 열어야 하는 파일은 `echo`이고, `hello`, `world`는 파일 로딩이 아니라 사용자 프로그램에 전달할 인자입니다.

`load()`가 성공하면 내부에서 기본 사용자 스택 페이지가 만들어지고, `intr_frame`의 `rsp`가 `USER_STACK`으로 준비됩니다.

그 다음 `build_user_stack_args(&_if, argc, argv, &argv_user_addr)`를 호출해서 실제 Argument Passing의 핵심인 사용자 스택 배치를 수행합니다.

---

## 5. 사용자 스택 배치: `build_user_stack_args()`

이 함수는 파싱된 인자들을 사용자 스택 위에 ABI가 기대하는 형태로 배치합니다.

처음에는 `sp`를 `user_if->rsp`로 시작합니다. 이 값은 `setup_stack()` 이후의 초기 스택 포인터입니다.

그리고 문자열을 뒤에서부터 복사합니다.

예를 들어 `echo hello world`라면 `world`, `hello`, `echo` 순서로 스택에 복사합니다. 스택은 높은 주소에서 낮은 주소로 자라기 때문에, 이렇게 역순으로 넣어야 최종적으로 문자열들이 자연스럽게 배치됩니다.

각 문자열을 넣을 때는 `strlen(argv[i]) + 1`로 널 종료 문자까지 포함한 길이를 계산하고, `sp`를 그만큼 낮춘 뒤 `memcpy()`로 복사합니다. 그리고 그 문자열이 사용자 스택 안에서 시작하는 주소를 `arg_addrs[i]`에 저장합니다.

여기서 `arg_addrs`도 지역 배열이 아니라 `palloc_get_page()`로 할당했습니다. 이유는 `ARG_MAX` 크기의 포인터 배열이 커널 스택을 많이 차지할 수 있기 때문입니다.

문자열을 모두 복사한 다음에는 `sp &= ~0x7`로 8바이트 정렬을 맞춥니다.

그 다음 `argv[argc]` 역할을 하는 NULL 센티널을 먼저 push합니다. 사용자 프로그램은 `argv[argc] == NULL`이라는 규칙을 기대하기 때문에 이 값이 반드시 필요합니다.

이후에는 아까 저장해둔 문자열 주소들을 다시 역순으로 push합니다. 이렇게 하면 메모리상에서는 `argv[0]`, `argv[1]`, `argv[2]` 순서로 읽히는 포인터 배열이 완성됩니다.

포인터 배열을 모두 넣은 뒤 현재 `sp`가 바로 사용자 스택 안의 `argv` 시작 주소입니다. 그래서 이 값을 `argv_user_addr`에 저장합니다.

마지막으로 fake return address인 0을 push하고, 최종 `sp`를 `user_if->rsp`에 반영합니다.

정리하면 스택에는 아래 순서의 정보가 만들어집니다.

```text
낮은 주소
  fake return address
  argv[0] 포인터
  argv[1] 포인터
  argv[2] 포인터
  NULL
  정렬 패딩
  "echo\0"
  "hello\0"
  "world\0"
높은 주소
```

말로는 복잡해 보이지만, 코드 기준으로는 세 가지 불변식만 지키면 됩니다.

1. 문자열은 널 종료 문자까지 포함해 사용자 스택에 복사한다.
2. `argv[argc]`는 반드시 NULL이다.
3. `argv_user_addr`는 커널 주소가 아니라 사용자 스택 안의 포인터 배열 시작 주소다.

---

## 6. 3단계: 레지스터 세팅

스택 배치가 끝나면 `set_user_entry_registers()`를 호출합니다.

64비트 System V ABI에서는 함수의 첫 번째 인자가 `RDI`, 두 번째 인자가 `RSI`에 들어갑니다.

따라서 `_start(argc, argv)`가 정상적으로 시작하려면 `RDI`에는 `argc`, `RSI`에는 사용자 스택 안의 `argv` 주소가 들어가야 합니다.

현재 구현에서는 `user_if->R.rdi = argc`, `user_if->R.rsi = argv_user_addr`로 이 값을 직접 설정합니다.

여기서 중요한 점은 `argv_user_addr`가 커널의 `argv` 배열 주소가 아니라는 것입니다. 커널 배열은 사용자 프로그램이 접근할 수 없습니다. 사용자 프로그램이 읽어야 하는 것은 아까 `build_user_stack_args()`에서 사용자 스택 안에 만든 포인터 배열입니다.

그래서 `set_user_entry_registers()`에서는 `argv_user_addr`가 사용자 가상 주소인지도 검사합니다.

그 다음 `validate_user_entry_frame()`에서 최종 `rsp`가 사용자 주소인지, `rdi` 값이 유효한 범위인지, `rsi`가 사용자 주소인지 마지막으로 확인합니다.

여기까지 성공하면 `file_name`, `cmd_line`, `argv` 같은 커널 임시 버퍼를 해제하고 `do_iret(&_if)`로 유저 모드에 진입합니다.

`do_iret()` 이후에는 커널 함수로 돌아오지 않기 때문에, 성공 경로의 마지막은 `NOT_REACHED()`가 됩니다.

---

## 7. 구현하면서 신경 쓴 부분

제가 구현에서 특히 신경 쓴 부분은 세 가지입니다.

첫 번째는 실패 경로의 자원 해제입니다.

`process_exec()`에서는 `file_name`, `cmd_line`, `argv`처럼 여러 페이지를 할당합니다. 중간 단계에서 실패하면 이미 할당한 페이지를 모두 해제해야 누수가 생기지 않습니다. 그래서 파싱 실패, 로드 실패, 스택 빌드 실패, 레지스터 세팅 실패마다 해제 경로를 명확히 넣었습니다.

두 번째는 커널 스택 사용량입니다.

PintOS에서는 thread 하나가 4KB 페이지를 쓰고, 그 안에 `struct thread`와 커널 스택이 같이 존재합니다. `char *argv[ARG_MAX]`, `char *arg_addrs[ARG_MAX]` 같은 배열은 각각 약 1KB 정도를 차지할 수 있어서, 호출이 깊어지면 `thread_current()`의 magic 값이 깨질 수 있습니다.

그래서 `argv`와 `arg_addrs`를 지역 배열이 아니라 페이지 할당으로 옮겼습니다.

세 번째는 단계별 책임 분리입니다.

`parse_command_line_args()`는 토큰화만 하고, `build_user_stack_args()`는 사용자 스택 구성만 하고, `set_user_entry_registers()`는 레지스터 설정만 합니다. 이렇게 나누면 테스트가 실패했을 때 어느 단계가 문제인지 좁히기 쉽습니다.

---

## 8. 테스트와 디버깅 관점

테스트는 `args-single`, `args-multiple`, `args-many`, `args-dbl-space`처럼 인자 개수와 공백 패턴이 다른 케이스를 기준으로 확인할 수 있습니다.

실패가 나면 저는 세 단계로 나눠서 봅니다.

첫 번째, 파싱 단계에서 `argc`와 `argv[]`가 기대한 순서인지 확인합니다.

두 번째, 스택 단계에서 문자열 주소와 포인터 배열이 올바르게 들어갔는지 확인합니다. 이때는 `hex_dump()`로 최종 `rsp` 주변을 보면 문자열, NULL 센티널, 포인터 배열 위치를 직접 확인할 수 있습니다.

세 번째, 레지스터 단계에서 `RDI`가 `argc`, `RSI`가 사용자 스택 안의 `argv` 주소인지 확인합니다.

이 세 단계 중 하나라도 틀리면 사용자 프로그램 입장에서는 `argv`를 읽는 순간 잘못된 주소를 따라가게 되고, 결국 페이지 폴트나 잘못된 출력으로 이어집니다.

---

## 9. 마무리

정리하면 Argument Passing은 단순히 문자열을 나누는 기능이 아니라, 사용자 프로그램이 시작되는 순간의 ABI 상태를 커널이 직접 만들어주는 작업입니다.

저희 구현에서는 `process_exec()`에서 입력을 검증하고 커맨드라인을 파싱한 뒤, `load()`로 실행 파일과 기본 스택을 준비하고, `build_user_stack_args()`로 사용자 스택 안에 문자열과 `argv` 포인터 배열을 구성했습니다.

마지막으로 `set_user_entry_registers()`에서 `RDI=argc`, `RSI=argv_user_addr`를 설정하고 `do_iret()`로 유저 모드에 진입합니다.

이 흐름을 통해 사용자 프로그램은 마치 일반 C 프로그램처럼 `main(argc, argv)`에서 인자를 읽을 수 있게 됩니다.
