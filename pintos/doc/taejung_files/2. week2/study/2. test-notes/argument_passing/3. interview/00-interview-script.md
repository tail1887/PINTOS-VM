# Argument Passing 면접 대본

## Q1: "Argument Passing이 뭐 하는 거예요?"

### 답변 (30초 - 기본 설명)

Argument Passing은 사용자 프로세스 시작 시 커맨드라인 인자를 C 호출 규약에 맞춰 전달하는 기능입니다.

예를 들어 `echo hello world`라고 실행하면, 커널이 "hello"와 "world"라는 두 인자를 사용자 스택에 올바른 주소로 배치한 후, `RDI`와 `RSI` 레지스터를 `argc=3`, `argv` 포인터로 세팅해서 `_start(argc, argv)` 함수가 C 표준대로 읽을 수 있게 만드는 것입니다.

이게 없으면 사용자 프로그램이 `argv`를 잘못된 주소에서 읽으려다 페이지 폴트로 즉시 종료됩니다.

---

## Q2: "그런데 구체적으로 어떤 단계를 거쳐요?"

### 답변 (2-3분 - 단계별 설명)

**3가지 단계로 나뉩니다.**

### 단계 1: 파싱 단계

커맨드라인 문자열 전체를 받아서 공백을 기준으로 토큰들을 분리합니다. `strtok_r()`을 사용해서 첫 토큰은 실행 파일명으로, 나머지는 인자들로 배열에 저장합니다. 

이 과정에서 중요한 건:
- 원본 문자열을 수정하면 안 되니까 복사본을 만든 후에 파싱한다
- 연속 공백은 하나의 구분자로 취급해야 한다

### 단계 2: 스택 구성 단계

파싱한 토큰들을 사용자 스택에 배치하는데, 스택이 위에서 아래로 자라는 구조이므로 문자열들을 역순으로 복사합니다. 

그 다음 8바이트 정렬을 맞춘 후, `argv[argc]` 위치에 NULL을 넣고, 문자열 주소들을 역순으로 포인터 배열로 만듭니다. 

이렇게 하면 최종적으로 사용자 코드가 `argv[0]`부터 정상 순서로 읽을 수 있게 됩니다.

### 단계 3: 레지스터 세팅

x86-64 호출 규약에 따라:
- `RDI` 레지스터에 `argc`를
- `RSI` 레지스터에 스택 안의 `argv` 포인터 배열 주소를
- `RSP`와 `RIP`를 설정해서 유저 모드로 진입시킵니다.

## Q3: "스택 구성 부분을 좀 더 자세히 설명해줄래요?"

### 답변 (1-2분 - 스택 레이아웃 설명)

네, 이 부분이 가장 중요합니다. 구체적인 예로 설명하겠습니다.

`echo hello world`를 실행한다고 하면, 최종 스택 모양은 이렇습니다.

```
[높은 주소]
  fake return address (0)     ← 이걸 마지막에 push
  argv[0] 포인터 (echo 주소)
  argv[1] 포인터 (hello 주소)
  argv[2] 포인터 (world 주소)
  NULL (argv[3])              ← 센티널
  [8-byte 정렬 패딩]
  "world\0"                   ← argc-1부터 역순 push
  "hello\0"
  "echo\0"
  rsp (현재 위치)
[낮은 주소]
```

포인트는 **문자열은 역순으로**, **포인터도 역순으로** 푸시하는 건데, 이렇게 하면 최종적으로 포인터 배열이 `argv[0]`, `argv[1]`, `argv[2]` 순서가 되는 겁니다.

그리고 주의할 점은 `argv`가 단순히 배열이 아니라 **스택 안의 주소**라는 거예요. 사용자 코드가 `argv[1]`을 읽으려면 먼저 `argv` 포인터가 가리키는 주소를 따라가서 거기서 `argv[1]`을 찾아야 합니다.

## Q4: "그런데 8바이트 정렬이 왜 필요하죠?"

### 답변 (30초 - 레지스터 정렬)

x86-64 ABI 규약에서 함수 호출 시점에 `RSP`가 16바이트 정렬되어야 한다고 정의했습니다.

fake return address를 푸시한 후의 `RSP` 주소가 정확히 16바이트 경계에 떨어지려면, 그 직전에 몇 바이트를 맞춰야 합니다. 이를 8바이트 정렬이라고 하는데, 이걸 맞추지 않으면 SSE 명령어 같은 고급 연산에서 segmentation fault가 납니다.

## Q5: "구현 중에 가장 어려웠던 부분이 뭐예요?"

### 답변 (1-2분 - 어려운 점과 해결)

**두 가지가 있었습니다.**

### 어려운 점 1: 스택 주소 계산

문자열들을 역순으로 푸시하면서 주소를 기록하고, 그 주소들을 다시 역순으로 푸시해야 하는데, 이 과정에서 인덱스를 잘못 관리하면 `argv[1]`이 `argv[2]` 값을 가리키는 식의 오프셋 오류가 생깁니다. 

디버깅할 때 `hex_dump()`로 스택 메모리를 직접 까서 확인했는데, 포인터들이 실제로 어디를 가리키는지 확인하는 게 중요했습니다.

### 어려운 점 2: 경계 케이스

`args-none` 같이 인자가 없는 경우와 `args-many` 같이 많은 인자가 있는 경우를 모두 처리해야 했는데, 특히 스택 페이지 경계(`USER_STACK - PGSIZE`)를 넘지 않도록 매번 체크하는 부분이 복잡했습니다. 

중간에 놓치면 테스트는 통과해도 엣지 케이스에서 crash가 났었거든요.

**해결 방법**: 매 push 연산마다 경계를 체크하고, 테스트를 하나씩 통과시키면서 각 단계에서 스택 덤프를 찍어서 검증하는 식으로 했습니다.

## Q6: "테스트는 어떻게 검증했어요?"

### 답변 (1분 - 테스트 전략)

**5가지 테스트를 순서대로 진행했습니다.**

```
args-none → args-single → args-multiple → args-many → args-dbl-space
```

각 테스트마다:
- **1단계 실패 시**: 파싱 부분만 확인 (`argc`, 토큰 배열 로그)
- **2단계 실패 시**: 스택 덤프로 문자열/포인터 위치 확인 (`hex_dump()`)
- **3단계 실패 시**: 레지스터 값 확인 (`RDI=argc?`, `RSI=argv_user_addr?`)

특히:
- `args-dbl-space`에서는 연속 공백이 제대로 처리되는지 확인
- `args-many`에서는 많은 인자가 들어왔을 때 스택이 오버플로우하지 않는지 확인

## Q7: "혹시 놓쳤던 부분이나 버그가 있었어요?"

### 답변 (1-2분 - 자기반성)

**네, 몇 가지가 있었습니다.**

### 버그 1: `argv[argc] = NULL` 센티널

처음에는 파싱 단계에서 이미 배열의 끝에 NULL을 넣는 줄 알았는데, 실제로는 스택 구성 단계에서 포인터 배열의 마지막에 NULL을 명시적으로 푸시해야 합니다. 

이걸 놓쳤을 때 `args-multiple`이 간헐적으로 실패했습니다.

### 버그 2: 8바이트 정렬

처음에는 `rsp = rsp - (rsp % 8)` 이렇게 했는데, 나중에 알고 보니 `rsp = (rsp - n) & ~7` 이런 식으로 정렬해야 합니다. 

이게 틀렸을 때는 간헐적인 crash가 났어서 추적하기 어려웠습니다.

### 버그 3: 실패 경로

초반에는 파싱 실패 후 자원 정리가 불완전했는데, 이러면 이후 테스트들이 메모리 누수로 간헐적으로 실패합니다. 

각 단계에서 실패할 때마다 명확한 cleanup 경로를 만드는 게 중요했습니다.

## Q8: "이 기능이 실제로 어디에 쓰여요?"

### 답변 (30초 - 실용성)

모든 현대식 운영체제에서 필수적입니다.

우리가 터미널에서 `./program arg1 arg2` 라고 실행할 때, 운영체제가 이 기능으로 arg1, arg2를 프로그램이 읽을 수 있게 준비합니다.

Linux의 `execve()` 시스템콜, Windows의 `CreateProcess()`도 모두 내부적으로 비슷한 메커니즘을 가지고 있습니다. 

그래서 이 구현을 이해하면 실제 OS가 어떻게 프로세스를 시작하는지 이해할 수 있습니다.

## Q9: "만약 정렬을 안 했다면 뭐가 될까요?"

### 답변 (30초 - 심화 질문)

SSE(Streaming SIMD Extensions) 명령어들이 16바이트 정렬된 주소를 기대하기 때문에, 정렬이 안 되면:
- 초기 접근 시에는 문제 없다가
- 라이브러리 함수가 SSE 명령어를 사용할 때 segmentation fault 발생
- 특히 복잡한 라이브러리(glibc 등)를 쓰는 대형 프로그램에서 잘 터진다

그래서 테스트 프로그램에서는 눈에 띄지 않다가 실제 프로그램에서 crash가 되는 식의 버그가 될 수 있습니다.

## Q10: "코드 리뷰 관점에서 가장 중요한 부분이 뭐예요?"

### 답변 (1분 - 코드 품질)

**경계 검사와 실패 경로입니다.**

### 경계 검사
매 push 연산 전에 `USER_STACK - PGSIZE` 하한을 체크해야 합니다. 이걸 놓치면 스택 오버플로우가 되어도 모르고 지나갑니다.

### 실패 경로
파싱/로드/스택/레지스터 중 하나라도 실패하면 부분적으로 준비된 상태에서 유저 모드로 진입하면 안 됩니다. 각 단계에서 실패했을 때 이전 단계의 자원을 정리하는 명확한 경로가 필요합니다.

### 주소 검증
`argv_user_addr`이 정말 사용자 가상 주소인지 (커널 주소가 아닌지) 확인해야 보안 이슈가 생기지 않습니다.

이 세 가지가 제대로 되어 있으면 안정적인 구현입니다.

## 🎯 30초 버전 (긴급 요약)

Argument Passing은 커맨드라인을 파싱해서 사용자 스택에 C 호출 규약대로 배치한 후, RDI/RSI 레지스터로 argc/argv를 전달하는 기능입니다. 

문자열과 포인터 배열을 역순으로 스택에 올리고, 8바이트 정렬을 맞춘 후, 유저 모드로 진입합니다. 

이 기능이 없으면 사용자 프로그램이 argv를 읽지 못해 즉시 페이지 폴트로 종료됩니다.

## 📝 추가 참고: 자주 들어오는 심화 질문

### "포인터 배열 순서가 왜 역순인가요?"
스택이 위→아래로 자라기 때문입니다. `argv[0]`을 첫 번째로 push하면 가장 높은 주소에 올라가는데, 사용자 코드가 배열 인덱스로 접근할 때는 낮은 주소부터 읽어야 하니까 역순으로 push합니다.

### "스택이 왜 낮은 주소로 자라나요? 마음대로 바꿀 수 없나요?"
좋은 질문입니다! 스택이 낮은 주소로 자라는 건 **하드웨어 설계** 때문이에요.

**하드웨어 설계의 이유**:
- x86 아키텍처에서 `push` 명령어는 `rsp`를 감소시키고, `pop`은 증가시킵니다
- 이는 메모리 레이아웃 설계에서 왔어요: 코드 영역은 낮은 주소, 힙은 높은 주소로 자라는데, 스택은 메모리 끝에서 시작해서 아래로 자랍니다
- 이렇게 하면 힙과 스택이 서로 만나면서 메모리를 효율적으로 사용할 수 있어요

**왜 바꿀 수 없나요?**
- ABI (Application Binary Interface)가 이렇게 정의되어 있어요
- 컴파일러, 라이브러리, OS가 모두 이 규약을 따릅니다
- Pintos도 x86 하드웨어 위에서 돌아가기 때문에 이 규약을 따라야 해요
- 마음대로 바꾸면 모든 프로그램이 깨집니다

**Pintos에서 왜 그대로 따르나요?**
- 실제 OS처럼 동작해야 하니까요
- 사용자 프로그램이 C 표준 라이브러리를 쓸 수 있게 하려면 스택 레이아웃을 맞춰야 합니다

결론적으로, 스택이 낮은 주소로 자라는 건 하드웨어/ABI 규약이고, 우리는 그 규약을 따라야 합니다. 마음대로 바꿀 수는 있지만, 그러면 실제 OS와 호환되지 않아요.

### "스택이 쌓인다는게 어짜피 낮은 주소부터 하나씩 추가하는거아니야? 왜 역순으로 추가하는거야?"
좋은 질문입니다! 스택의 기본 동작을 먼저 이해해야 해요.

**스택의 기본 동작**:
- 스택은 **높은 주소에서 낮은 주소로 자랍니다**. push하면 `rsp`가 감소합니다.
- 예를 들어, `USER_STACK`이 0x1000이라고 하면, push할 때마다 주소가 0x1000 → 0xFF8 → 0xFF0으로 낮아집니다.

**문제점**: 사용자 코드가 `argv`를 읽을 때는 **낮은 주소부터 읽습니다**.
- `argv[0]`이 가장 낮은 주소에 있어야 `argv[1]`, `argv[2]` 순서대로 읽을 수 있습니다.

**그래서 역순 push가 필요한 이유**:
`echo hello world` 예시로 보면:

**직접 순서로 push하면 (문제 발생)**:
```
push "echo"     → rsp = 0xFF0, argv[0] 주소 = 0xFF0
push "hello"    → rsp = 0xFE8, argv[1] 주소 = 0xFE8  
push "world"    → rsp = 0xFE0, argv[2] 주소 = 0xFE0
```
이렇게 하면 포인터 배열도 `argv[0]`부터 push하므로:
```
포인터 배열: [0xFF0, 0xFE8, 0xFE0] → argv[0] = 0xFF0 (가장 높은 주소)
```
사용자 코드가 `argv[0]`부터 읽으려면 가장 높은 주소부터 읽게 되는데, 이는 잘못된 순서입니다.

**역순 push로 해결**:
```
push "world"    → rsp = 0xFF0, addr[2] = 0xFF0
push "hello"    → rsp = 0xFE8, addr[1] = 0xFE8
push "echo"     → rsp = 0xFE0, addr[0] = 0xFE0
```
이제 포인터 배열을 역순 push:
```
포인터 배열: [0xFE0, 0xFE8, 0xFF0] → argv[0] = 0xFE0 (가장 낮은 주소) ✅
```

**결론**: 스택이 낮은 주소로 자라기 때문에, 사용자 코드의 읽기 순서(낮은 주소부터)를 맞추려면 역순으로 데이터를 배치해야 합니다.

### "코드에서는 push 명령어를 안 쓰는데 왜 스택이 아래로 자란다고 하는 거야?"
좋은 지적입니다! Pintos는 **커널 모드**에서 실행되기 때문에 하드웨어 push/pop 명령어를 직접 사용할 수 없어요.

**커널 모드 vs 유저 모드 차이**:
- **유저 모드**: `push rax` 명령어로 하드웨어가 자동으로 `rsp -= 8; *(rsp) = rax` 수행
- **커널 모드**: 하드웨어 명령어 제한적, 직접 메모리 조작해야 함

**Pintos에서 스택 조작하는 방법**:
```c
// 하드웨어 push 대신 이렇게 함
sp -= len;              // rsp 감소 (스택 자람)
memcpy((void *) sp, argv[i], len);  // 데이터 복사
```

**왜 "스택이 아래로 자란다"라고 하는가?**
- 개념적으로는 맞아요: 스택 포인터가 감소하면서 메모리가 채워짐
- 실제로는 소프트웨어적으로 `sp -=` 연산으로 구현

### "이 메모리 구성이 한페이지마다 이렇다는거야? 그러면 각페이지들의 주소가 다 같은거야?"
좋은 질문입니다! 제가 설명한 메모리 구성은 **전체 프로세스의 가상 주소 공간 레이아웃**을 보여주는 거예요. 각 페이지마다 동일한 구조가 적용되는 게 아닙니다.

**메모리 구성의 의미**:
- **프로세스별 레이아웃**: 각 프로세스는 독립적인 가상 주소 공간을 가짐
- **영역별 역할 분담**: 코드/데이터는 낮은 주소, 힙은 중간, 스택은 높은 주소
- **페이지 단위 할당**: 각 영역은 4KB 페이지들로 구성되지만, 구조는 다름

**각 페이지들의 주소가 같은가?**
- **아니요, 각 페이지의 가상 주소는 다릅니다!**
- 예를 들어:
  - 코드 페이지: 0x08048000 ~ 0x08049000
  - 데이터 페이지: 0x08049000 ~ 0x0804A000  
  - 힙 페이지: 동적으로 할당 (예: 0x20000000 ~ 0x20001000)
  - 스택 페이지: USER_STACK 근처 (예: 0x4747F000 ~ 0x47480000)

**왜 주소가 다른가?**
- **가상 메모리 시스템**: 각 프로세스는 0~2^64 범위의 독립된 주소 공간을 가짐
- **페이지 테이블**: 각 페이지가 물리 메모리의 다른 위치에 매핑됨
- **효율성**: 코드와 데이터를 분리해서 캐시 효율을 높이고, 보안을 강화

**Pintos에서 확인하는 방법**:
```c
// 각 영역의 주소 범위
#define USER_STACK 0x47480000          // 스택 시작
#define PHYS_BASE 0x8004000000         // 커널 시작
// 코드/데이터는 ELF 헤더에서 로드 시 결정
```

결론적으로, 메모리 구성은 "전체 맵"이고, 각 페이지는 그 맵 안에서 서로 다른 주소를 가집니다. 한 페이지마다 동일한 구조가 아니라, 역할에 따라 다른 주소에 배치됩니다.

### "그러면 저 모든 주소들이 다 가상주소를 가지고 있는거야?"
네, 맞습니다! Pintos의 **모든 메모리 주소는 가상 주소**입니다.

**Pintos 메모리 모델**:
- **가상 메모리 시스템**: 물리 메모리와 분리된 가상 주소 공간 사용
- **페이지 테이블**: 가상 주소를 물리 주소로 변환 (MMU가 담당)
- **프로세스별 격리**: 각 프로세스는 독립적인 가상 주소 공간 가짐

**주소 범위**:
- **사용자 공간**: `0x0000000000000000` ~ `0x8000000000000000` (가상 주소)
- **커널 공간**: `0x8000000000000000` ~ `0xFFFFFFFFFFFFFFFF` (가상 주소)
- **물리 메모리**: 실제 RAM 주소, 페이지 테이블로 매핑

**왜 가상 주소인가?**
- **보안**: 프로세스들이 서로의 메모리를 직접 접근 못 함
- **편의성**: 연속된 주소 공간처럼 사용 가능 (실제 물리 메모리는 단편화됨)
- **효율성**: 페이지 단위로 메모리 관리, 필요할 때만 물리 메모리 할당

**Pintos에서 확인**:
```c
// 사용자 주소 검증 함수
bool is_user_vaddr(const void *va) {
    return va < (void *) LOADER_KERN_BASE;  // 0x8004000000
}

// 커널 주소 예시
#define PHYS_BASE 0x8004000000  // 커널 가상 주소 시작
```

결론적으로, 코드/데이터/힙/스택의 모든 주소는 가상 주소이고, 페이지 테이블을 통해 물리 메모리에 매핑됩니다.

### "아 그러면 물리 메모리를 직접 참조하는 게 아니구나"
네, 맞습니다! Pintos는 **가상 메모리 시스템**을 사용하기 때문에 물리 메모리를 직접 참조하지 않습니다.

**가상 메모리 작동 방식**:
- **모든 주소는 가상 주소**: 코드에서 사용하는 주소는 모두 가상 주소
- **MMU 변환**: 메모리 액세스 시 하드웨어(MMU)가 페이지 테이블을 통해 가상 주소를 물리 주소로 변환
- **커널도 가상 주소 사용**: 커널 공간도 가상 주소 (PHYS_BASE부터 시작)

**왜 직접 참조하지 않는가?**
- **보안**: 프로세스들이 물리 메모리를 직접 건드리지 못하게 함
- **편의성**: 연속된 주소 공간처럼 사용할 수 있음 (실제 물리 메모리는 단편화됨)
- **멀티태스킹**: 각 프로세스가 독립적인 주소 공간을 가질 수 있음

**Pintos에서 물리 메모리 액세스**:
```c
// 물리 메모리 직접 액세스 (커널 전용)
#define PHYS_BASE 0x8004000000
void *paddr_to_kvaddr(uintptr_t paddr) {
    return (void *) (paddr + PHYS_BASE);  // 물리 → 커널 가상 주소 변환
}
```

결론적으로, 사용자 프로그램은 물리 메모리를 전혀 모르고 가상 주소만 사용합니다. 물리 메모리 관리는 전적으로 커널의 몫입니다.
- 결과적으로는 하드웨어 push와 동일한 효과

결론적으로, Pintos는 커널 모드 제약 때문에 직접 메모리 조작하지만, 개념적으로는 x86 스택 규약을 따르고 있어요.

### "그러면 힙도 같이 만드나?"
좋은 질문입니다! Pintos에서는 Argument Passing 단계에서 **힙을 별도로 만들지 않습니다**.

**Pintos에서 힙 처리 방식**:
- **즉시 할당**: `setup_stack()`에서 스택 페이지만 미리 할당
- **지연 할당**: 힙은 사용자 프로그램이 `malloc()` 호출 시 page fault 발생 → 그때 VM 시스템이 힙 페이지를 동적으로 할당
- **현재 VM 버전**: `#ifdef VM` 블록에서 supplemental page table이 힙 관리를 담당

**왜 Argument Passing에서 힙을 안 만드는가?**
- **필요성**: Argument Passing은 프로그램 시작 시 최소한의 스택만 필요
- **효율성**: 힙은 프로그램 실행 중에 필요한 만큼만 할당하는 게 효율적
- **단순성**: 스택만으로도 `args-*` 테스트 통과 가능

**실제 OS와 비교**:
- Linux: `brk()` 시스템콜로 힙 초기 크기 설정
- Pintos: page fault 기반 lazy allocation으로 더 유연하게 힙 관리

결론적으로, Pintos는 스택은 즉시 할당하고 힙은 필요할 때 할당하는 lazy 방식입니다.

### "기본적인 메모리의 구성이 어떻게 되는거야?"
좋은 질문입니다! Pintos의 메모리 구성은 다음과 같습니다.

**전체 메모리 레이아웃 (x86-64)**:
```
[높은 주소 - 커널 공간]
0xFFFFFFFFFFFFFFFF  ┌─────────────────────┐
                     │     커널 코드       │
                     │     커널 데이터     │
                     │     커널 힙         │
0x800000000000       ├─────────────────────┤ ← 커널/사용자 경계
                     │                     │
                     │   (사용 불가 영역)   │
                     │                     │
0x400000             ├─────────────────────┤ ← 사용자 공간 시작
                     │     사용자 힙       │ ← 동적 할당 (malloc)
                     │     (빈 공간)       │
                     │                     │
USER_STACK(0x47480000)├─────────────────────┤
                     │   사용자 스택       │ ← Argument Passing에서 설정
                     │   (argv, 인자들)    │
0x4747F000          ├─────────────────────┤ ← 스택 페이지 시작
                     │   사용자 데이터     │ ← ELF에서 로드
                     │   사용자 코드       │ ← ELF에서 로드
0x08048000          ├─────────────────────┤ ← 프로그램 시작
                     │   (빈 공간)         │
0x00000000          └─────────────────────┘
[낮은 주소]
```

**Argument Passing에서 설정하는 부분**:
- **스택 영역**: `USER_STACK` 근처에 스택 페이지를 할당하고 인자들을 배치
- **코드/데이터**: `load()` 함수에서 ELF 파일을 읽어 메모리에 로드
- **힙**: VM 시스템에서 필요할 때 동적으로 할당

**주요 특징**:
- **커널/사용자 분리**: 0x800000000000를 경계로 분리
- **페이지 기반**: 모든 메모리 할당이 4KB 페이지 단위
- **가상 메모리**: 물리 메모리와 분리된 가상 주소 공간

이렇게 구성하면 각 프로세스가 독립적인 메모리 공간을 가지면서도 효율적으로 메모리를 사용할 수 있습니다.

### "argv와 argv_user_addr의 차이가 뭐예요?"
- `argv`: 커널 공간의 배열 주소
- `argv_user_addr`: 사용자 스택 안에 배치된 포인터 배열의 주소 (사용자 가상 주소)

사용자 코드가 읽을 수 있는 건 `argv_user_addr`뿐입니다.

### "왜 fake return address를 push하나요?"
유저 프로그램이 `_start()` 종료 후 `ret` 명령어를 실행할 때 스택에서 반환 주소를 팝합니다. 이 시점에 유효한 주소가 없으면 crash가 되니까, 미리 더미 값을 놓습니다.
